Пояснения для внешних разработчиков (чтобы экспортировать данные так, чтобы можно было идемпотентно обогащать данные из выхода на вход в циклах обработки): "EkceloFoto-формат данных (для парсеров/визуализаторов) index.html . Внутреннее представление сессии — массив kmlLayers[], где каждый слой = {filename, docName, kmlText, parsedData, cadPlacemarks[], layerGroup, handle}. kmlText — это полный KML 2.2 XML (один <Document> с дочерними <Style> и <Placemark>); это canonical-форма данных, через которую идёт обмен с внешним миром. Каждый кадастровый/контурный объект, созданный в приложении, хранит готовые XML-строки в cadPlacemarks[i].parts = {style, placemark} и регенерируется в kmlText через _rebuildLayerKMLText(layer) после любого изменения геометрии. Импорт чужого KML/KMZ проходит через parseKML(xmlText, filename) → renderKMLOnMap(), при этом одинаковые placemark'и из разных файлов схлопываются в один по композитной подписи lowercase(trim(name)) || lowercase(trim(desc.slice(0,80))) (геометрия НЕ участвует — здание внутри своего ЗУ имеет те же координаты, но это разные объекты), так что повторная загрузка той же выгрузки не задваивает данные. Экспорт через «📤 Экспорт данных»: KML — байт-в-байт re-emission kmlText всех слоёв с photo-placemark'ами как <Point> в hex-цвете ff47ffe8; KMZ — то же + изображения в images/ папке архива; XLSX — два листа («Фото», «Объекты») с фиксированными колонками. Идемпотентность round-trip держится на трёх правилах: (1) сохраняйте <Style id> стабильными между выгрузками, (2) для созданных в EkceloFoto объектов <name> содержит кадастровый номер в формате NN:NN:NNNNNNN:NN или человекочитаемый ярлык — это ключ дедупликации, (3) не модифицируйте <description> свободным текстом в начале (первые 80 символов используются как fallback-ключ для дедупа placemark'ов с пустым <name>). FSA-handles файлов кешируются per-session — приложение перезаписывает тот же файл без диалога, если он был открыт через File System Access API в этой же сессии (cross-session — handle утерян, открывается picker). Чего внутренний формат НЕ хранит и НЕ гарантирует: оригинальный whitespace/комментарии/<ExtendedData> чужого KML могут быть не сохранены при re-emission; геометрии в MultiGeometry упрощаются до Polygon/MultiPolygon; высоты <altitude> для <Point> фотографий принудительно ставятся в 0 (clampToGround). Если ваш парсер должен сохранить кастомные расширения чужого KML — добавьте их через <ExtendedData><Data name="…"> ВНУТРЬ <Placemark>, но не как сиблингов на уровне <Document>."
