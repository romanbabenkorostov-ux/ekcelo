# Нестабильность повторного парсинга ЕГРН — диагностика и фикс

> Симптом: повторный парсинг одних и тех же выписок даёт diff'ы (поле «меняется»),
> хотя источник не менялся. Разобрано по вложениям (выписка XML + Приказ о составе
> выписки ЕГРН). Дата: 2026-06-05.

## Нормативная основа
Состав выписки из ЕГРН детерминирован Приказом Росреестра (форма/состав сведений);
**XML — машиночитаемая авторитетная форма**. Значит вариативность между прогонами на
одном источнике — это **дефект парсера/пайплайна, а не данных**.

## Корневые причины

### 1. Один отчёт обрабатывается дважды: PDF и XML
Во входной папке `EGRN` лежат и `*.pdf`, и `*.xml` **одного и того же отчёта**
(126 pdf + 17 xml). Парсер обрабатывает оба → второй источник «диффится» против
первого. При этом:
- **Классификация ограничений расходится.** `pdf_parser._parse_object_restrictions`
  и `xml_parser._parse_xml_object_restrictions` по-разному относят ограничение к
  `czuit_zone` / `okn_territory` (разный исходный текст PDF vs структура XML) →
  diff `object_restrictions: czuit_zone → okn_territory`.
- **Порядок элементов в JSON-массиве** мог различаться → строковое сравнение в differ
  давало ложный diff.

### 2. Обогащённые поля диффятся против сырого парса
`resolve_room_parent` алгоритмически заполняет `parent_cad_number` (лог: «обновлено
15 помещений»). Сырой повторный парс той же выписки `parent_cad_number` не содержит →
diff `parent_cad_number: 61:44:0040713:298 → —`.

## Что починено в этом архиве
- **`differ.py` — каноническое сравнение JSON-массивов** (`object_restrictions`,
  `permitted_uses`, `land_cad_numbers`, `old_numbers`, `nested_objects`): порядок
  элементов не важен; пустой массив ≡ `None`. Реальная смена значения (тип
  ограничения) по-прежнему фиксируется. Замок: `tests/test_differ_json_canonical.py`.
- **`interactive.py` — промпты `y/n` с дефолтом из рекомендации.** Enter = рекомендация
  парсера (пусто в БД → принять; нет в выписке/различие → оставить БД). Общий выбор:
  `[e/r/n/s/d/q]`, Enter = `enrich` (непедеструктивно). Исправлен баг `д`→`d`.
- **`make_demo_block2.py`** — спрашивает про существующую БД (оставить/пересоздать),
  флаг `--force`, безопасное поведение без TTY.

## Рекомендации (follow-up, не в этом архиве)
1. **Дедуп входа:** на один `cad_number` + `extract_number` обрабатывать ОДИН
   источник; приоритет **XML > PDF** (авторитетная форма). Снимает корень №1.
2. **Выровнять классификаторы ограничений** PDF↔XML: единый словарь ключевых слов
   (`культурного наследия`→ОКН, ЗОУИТ-маркеры→czuit_zone) в общем модуле.
3. **Провенанс полей:** не диффить алгоритмически обогащённые поля
   (`parent_cad_number`) против сырого парса — помечать источник (`raw`/`enriched`)
   и сравнивать однородное с однородным.
