# Ментальный отчёт v5: высоты, EXIF, KMZ и корневая «Не_распределено»

**Дата:** 2026-05-17 (обновление v4)
**Что добавлено в этой итерации (v5):**
- Папка `Не_распределено/` перенесена **в корень болванки** (раньше была
  внутри `Фотографии/`).
- Меню 3 скрипта `pirushin_sosn_rocha_07_init_project_v1.py` разносит
  её содержимое по структуре проекта **для любых типов файлов**

- При сортировке JPG в EXIF дописывается привязка к КН/категории
  (`ImageDescription`, `UserComment`-JSON), но **оригинальные** даты
  съёмки (`DateTime*`) и GPS не затрагиваются. `mtime/atime` файлов
  сохраняются через `os.utime`.

**Что осталось из v4:**
- Высоты по русской системе для уровней зданий (3 м на этаж).
- Объёмные контуры зданий (`height_m`, `_geometry.top_z_meters`, `extrude=true`).
- Высоты оборудования по уровню (`z_meters`).
- Чтение EXIF из JPEG (GPS lat/lon/altitude/bearing, дата, камера).
- Высота фото: либо z уровня объекта, либо EXIF altitude.

## 1. Логика высот (compute_level_z)

| Тип уровня | Формула | Пример (4-этажка) |
|---|---|---|
| Подвал K | `−K × 3` | подвал 1 = −3, подвал 2 = −6 |
| Полуподвал | `−2` | −2 |
| Цокольный этаж | `−2` (= полуподвал, по соглашению) | −2 |
| Техническое подполье | `−1` | −1 |
| Этаж K (надз.) | `(K-1) × 3` | этаж 1 = 0, этаж 4 = 9 |
| Антресоль K | `(K-1) × 3 + 2` | антресоль 1 этажа = 2, антр. 2 этажа = 5 |
| Антресоль (без №) | `(N-1) × 3 + 2` | антресоль для 4-этажки = 11 |
| Мансарда / Чердак | `N × 3` | 12 |
| Технический этаж сверху | `(N+1) × 3` | 15 |

Распознавание типа — по `level.type` (case-insensitive). Конкретный номер
этажа — из `label` («Уровень N. Этаж K»).

**Тест на 12 кейсах:** 12/12 прошло (см. вывод unit-теста).

## 2. Объёмная геометрия зданий

Для каждого ОКС с этажностью:
```jsonc
"height_m": 12.0,           // floors_above × 3
"depth_m":  0.0,            // floors_below × 3 (без знака)
"_geometry": {
  "points_normalized": [...],     // контур ЗУ/здания (есть в enriched)
  "center":             [lon, lat],
  "bbox":               [...],
  "extrude":            true,      // ← готов к 3D-extrude в KMZ
  "bottom_z_meters":    -0.0,      // -depth_m
  "top_z_meters":       12.0       // +height_m
}
```

KMZ-экспортёр (отдельный скрипт в будущем) делает:
- из `points_normalized + center + bbox` восстанавливает абсолютные lon/lat;
- строит многогранник: нижняя грань z=`bottom_z_meters`, верхняя
  z=`top_z_meters`, боковые грани по контуру.

## 3. Высоты оборудования

```python
for eq in equipment:
    z_vals = [level_z[l] for l in eq.links.level_ids if l in level_z]
    if z_vals:
        eq["z_meters"]     = min(z_vals)
        eq["z_meters_max"] = max(z_vals) + 3   # для оборудования, охватывающего несколько этажей (лифт)
```

Если оборудование привязано к нескольким уровням (например, лифт на
этажах 1-4) → `z_meters=0`, `z_meters_max=12` (вся высота шахты).

## 4. EXIF из JPEG (Pillow, опционально)

`read_exif(path)` возвращает (как `exifr` в JS-просмотрщике):
```python
{
  "gps_lat":        47.21875,
  "gps_lon":        39.70893,
  "gps_alt":        15.5,         # м над уровнем моря
  "gps_bearing":    180.0,        # GPSImgDirection
  "datetime_taken": "2026:03:15 14:30:00",
  "camera_make":    "...",
  "camera_model":   "..."
}
```

Поля **точно совпадают** с тем, что читает фронт-просмотрщик через
`exifr` (видно в `index.html`: `e.latitude`, `e.GPSAltitude`,
`e.GPSImgDirection`, `e.DateTimeOriginal`).

**Без Pillow:** функция возвращает `{}`, секция `exif` отсутствует, но
скрипт продолжает работать (высота берётся только из этажей).

## 5. Алгоритм `photos[].z_meters`

```
if is_unassigned (фото лежит в корневой Не_распределено/):
    z_meters = None         # ← по требованию: без обработки высотами
elif есть привязанный cadastre с единственным уровнем:
    z_meters = level.z_meters   # z_source = "level"
elif exif.gps_alt:
    z_meters = exif.gps_alt     # z_source = "exif_gps_alt"
else:
    z_meters = None
```

EXIF читается **всегда** — даже для `is_unassigned`. Это позволяет
пользователю перенести фото в нужную папку без потери метаданных
(координаты, дата, камера).

## 6. Папка `Не_распределено/` (v5: в корне болванки)

**Расположение (изменено в v5):** папка лежит в **корне болванки**,
рядом с `ОСВ/`, `Фотографии/`, `Выписки_PDF/`, `json/`, `XLSX/`, `DB/`
и др. Раньше (v4) она лежала внутри `Фотографии/Не_распределено/` —
это было неверно: пользователь складывает туда не только фото, но и
выписки ЕГРН в PDF/XML, договоры в DOC, реестры в XLS, базы `.db`,
сохранённые HTML-отчёты, KML-файлы.

**Создание (`make_tree` в `pirushin_sosn_rocha_07_init_project_v1.py`):**
```python
UNSORTED_DIR = "Не_распределено"
TXT_MD_DIR   = "txt-md"
DIRS = ["_data", "ОСВ", "xlsx", "db", "kmz-kml", "html", "doc",
        TXT_MD_DIR, "json", "json/objects", "Бизнес_единицы",
        "Документы_JPG", "Выписки_PDF", ..., UNSORTED_DIR,
        "Фотографии", "Фотографии/Недвижимость", ...]
```

**Регистр корневых служебных папок (v6):** `db/`, `html/`, `xlsx/`,
`kmz-kml/` — в нижнем регистре, согласно общепринятому соглашению для
служебных каталогов. Тематические русские папки (`ОСВ/`, `Фотографии/`,
`Выписки_PDF/`, `Документы_JPG/`, `Бизнес_единицы/`) остаются как были.
Добавлена `txt-md/` для текстовых заметок (`.txt`, `.md`).

**Разнесение (меню 3 скрипта 07):** `scan_unsorted(root)` + `action_sort`.
По расширению файла маршрутизирует:

| Расширение | Куда переносится |
|---|---|
| `.pdf` | `Выписки_PDF/<подпапка по classify_pdf>/` (ЕГРН / ЕГРЮЛ-ЕГРИП / Свидетельства_о_праве / Технические_паспорта / Техпланы / Прочее) |
| `.xml` | та же подпапка, что у парного PDF; иначе `Выписки_PDF/ЕГРН/` |
| `.doc`, `.docx` | `doc/` |
| `.xls`, `.xlsx` | `xlsx/` |
| `.db` | `db/` |
| `.html`, `.htm` | `html/` |
| `.json` | `json/` |
| `.kml`, `.kmz` | `kmz-kml/` |
| `.txt`, `.md` | `txt-md/` |
| `.sig`, `.asc`, `.p7s` | `Выписки_PDF/Прочее/` (ЭП-подписи рядом с выпиской) |
| `.jpg`, `.jpeg` | `Фотографии/Недвижимость/<категория>/<КН>/<sem-цепочка>/` |

Для JPG приоритет определения КН (v6, после рефакторинга
`build_dump_index` + `_find_cn_for_photo`):

1. **КН в цепочке родительских папок** — в трёх формах: с двоеточиями
   (`61:44:0050706:31`), с подчёркиваниями (`61_44_0050706_31`), плоский
   12-15 цифр подряд (`61440040713308`).
2. **КН в братском PDF/XML — поиск во ВСЕХ родительских папках вверх до
   `Не_распределено/`** (`iterdir`), не только в той же папке, где
   лежит JPG. Это критично для реальных дампов, где PDF и XML лежат на
   2-3 уровня глубже фото, в подпапке `Документы/<КН-папка>/*.pdf`.
3. **Адресная папка из индекса дампа** (`build_dump_index.addr2cn`):
   первый проход по дампу находит все папки, в названии которых есть
   КН (`61440040713343_квартира_...`), и связывает родительскую
   НЕ-контейнерную папку с этим КН. Например подпапка
   `Ростов-на-Дону_БСадовая_111/Документы/61440040713343_квартира_…/`
   делает `Ростов-на-Дону_БСадовая_111` «адресной папкой» КН
   `61:44:0040713:343`. Все фото внутри неё привязываются к этому КН.
4. **Адресный матчинг любой родительской НЕ-контейнерной папки** по
   `egrn_*.json` (если такие выгрузки лежат в дампе) или
   `json/objects/<КН>.json` — ≥ 2 общих токена адреса. При нескольких
   КН по одному адресу — фото остаётся в источнике, в лог пишется
   список кандидатов.

**Контейнерные папки** (`Документы`, `output`, `Коммерческие аналоги`,
`Фото`, `Foto`, `Архив`, `tmp`, `src` и любые с семантической меткой
типа `Состояние внутри`/`Вид от окна`/`Кухня`/…) при поиске
«адресной» родительской папки пропускаются.

Категория объекта (`Земельные_участки`/`Строения`/`Сооружения`/
`Помещения`/`ОНЗ`) определяется из `idx['types']` по полю `object_type`:

- из таблиц `building_objects` и `land_objects` в `egrn_*.json` дампа
  (для `land_objects` дефолт = `land`);
- из паспортов `json/objects/<КН>.json` (если они уже созданы);
- из имени подпапки с КН (`…_квартира_…` → room, `…_жилой_дом_…` →
  building, `…_зу_…` → land), это **новый источник в v6** — он
  даёт категорию даже когда ЕГРН-выгрузка по этому КН ещё не
  обработана;
- из полного текста пути с учётом сокращений и опечаток
  (`ЗУ/з.у./земучасток/зучесток` → land, `квартира/подавал` → room,
  `жилой дом/здан` → building, `сооруж` → structure, `ОНЗ/незаверш` →
  ons).

**Источники адресов** для матчинга (`idx['addrs']`):
- `json/objects/<КН>.json` болванки (`load_objects_index`),
- все `egrn_*.json` внутри дампа `Не_распределено/`
  (`_load_egrn_dump` читает таблицы `building_objects`/`land_objects`,
  достаёт `cad_number` + `address`).

Семантические подпапки нормализуются по словарю: `Вход_в_объект`,
`Общий_вид_адреса`, `Прилегающая_территория`, `Состояние_внутри`,
`Состояние_отделки`, `Состояние_объекта`, `Подвал`, `Чердак`,
`Вид_от_окна`, `План_эвакуации`, `Планы`, `Экспликация`, `Кухня`,
`Санузел`, `Фасад`, `Литер_<буква>`, `Кв_<номер>`, `Этаж_<N>`, `Аналоги`,
`Окружение_квартала`, `Документы_фото`. **Полная исходная иерархия меток
сохраняется** в целевом пути.

**Поведение при подтверждении плана:**
1. `shutil.move(src, dst)` — перенос файла.
2. `os.utime(dst, (atime, mtime))` — восстановление времён файла,
   которые мог сдвинуть `move`.
3. Только для JPG, если определён КН — `annotate_photo_exif`:
   - дописывает `0th.ImageDescription` (старое сохраняется через ` | `);
   - пишет `Exif.UserComment` = JSON `{app, kind:"photo", cad, category,
     semantic, source, migrated_at}`;
   - **НЕ ТРОГАЕТ** `0th.DateTime`, `Exif.DateTimeOriginal`,
     `Exif.DateTimeDigitized`, весь блок `GPS` (lat/lon/alt/bearing) и
     любые другие оригинальные EXIF-тэги (`Software`, `Make/Model`,
     `Orientation`, `ExposureTime`, …);
   - после `piexif.insert` повторно вызывается `os.utime`, чтобы запись
     EXIF не сдвинула mtime.

**Распознавание `is_unassigned` (для совместимости с прежним фронтом):**
фото считается «нераспределённым», если оно лежит в корневой
`Не_распределено/` болванки. Папки с тем же именем внутри других
категорий (если такие появятся) не считаются спец-папкой.

## 9. Что НЕ сделано в этой версии (намеренно)

1. **Парсинг подсказки уровня из имени папки `Этаж_2`** — пока
   используется только при поиске КН-маски. Если у объекта несколько
   уровней, фото с пути `…/Этаж_2/…` остаётся без точного `level_id`.
   Это можно усилить отдельной функцией `_resolve_level_from_path`.
2. **KMZ-экспорт** — отдельный скрипт. JSON содержит всё нужное.
3. **Высота над абсолютным нулём (над уровнем моря)** — не
   используется. `z_meters` всегда относительно «уровня 0» здания.
   EXIF altitude хранится отдельно в `exif.gps_alt`.
4. **Дополнительные форматы EXIF** (HEIC, RAW) — Pillow умеет HEIC при
   установке `pillow-heif`, остальные требуют отдельных библиотек.
   Это вне scope текущей задачи.
5. **Хэш-дедупликация при коллизии имён** в меню 3 — сейчас при
   совпадении пути назначения файл просто пропускается (идемпотентно).
   Если у вас в одной папке-цели окажутся разные файлы с одним именем —
   останется второй в `Не_распределено/`. Можно усилить SHA-256 + суффикс.

## 10. Совместимость

- **Обратная совместимость JSON**: новые поля (`height_m`, `z_meters`,
  `exif`, `is_unassigned`) — все optional. Старые потребители (граф 04,
  фото-отчёт 06) продолжают работать.
- **Изменение расположения `Не_распределено/`** (v4 → v5): прежняя
  папка `Фотографии/Не_распределено/` больше не создаётся
  `make_tree`. Если на диске уже есть проекты с прежним размещением —
  один раз вручную переместите её в корень болванки (или просто
  переименуйте: `move Фотографии\Не_распределено .\Не_распределено`).
- **Pillow / piexif** — опциональные зависимости. Без них работает всё,
  кроме чтения и дописывания EXIF (в меню 3 будет жёлтый warning, файл
  всё равно перенесётся).
