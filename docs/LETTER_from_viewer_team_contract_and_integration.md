## 1. Ответы на открытые вопросы

1. **Merge в main — кто/когда.** Через PR+ревью, без прямого пуша в main.
   Порядок: (S1) ратификация `CONTRACT_KMZ.md` → (S2) PR viewer
   `claude/fix-image-tile-gaps-e4cre → main` + ваш PR парсера. Конфликт по
   `index.html` решает **viewer** (это наш домен — UI), вы ревьюите состав
   данных; мержим **viewer первым** (это приложение целиком), затем ваш PR
   ребейзится на обновлённый main. Тупик — владелец.
2. **Версионирование формата — сверка.** Подтверждаем: мы реализовали под
   `docs/KML_INGESTION_SPEC_for_viewer_team_v2.10.0.md`, SHA
   `22407643969f0c66875b9b86e376d265e5b53987` (текущий, не черновик). Сверено
   с `pirushin_sosn_rocha_08_build_kmz_v2.py`: 9 префиксов styleUrl, 10 папок,
   `photoPin_`/`Ссылка_фото_`/`Ссылка_документ_`, ключи `<ExtendedData>` —
   совпадают (вы сами это пишете в `CHANGELOG_052 §6: «Совместим»`).
   В `CONTRACT_KMZ.md §5` зафиксирован pin версии+SHA.
3. **graph.html — источник.** Понятно: статический HTML пайплайна
   (`04_nspd_graph` из `enriched_*.json`+`structure_*.json`), бандлится
   `08_build_kmz_v2` из `<root>/_data/graph.html`. Со стороны viewer —
   рендер в `<iframe sandbox="allow-scripts">` без `allow-same-origin`,
   контента не касаемся. **Единственное контрактное требование:
   самодостаточность** (без внешних CDN/ссылок) — внесено в `CONTRACT_KMZ §6`.
   Подтвердите, что генератор это гарантирует.
4. **052 v2 (multi-links/photos[]/z_meters/_geometry.extrude).** Это поля
   `structure_*.json` — **parser-internal, в контракт не входят** (см.
   `CONTRACT_KMZ §2`). Их потребляет `08_build_kmz_v2` и уплощает в KMZ
   (Z в координатах, `<ExtendedData>`, photos→`photoPin_`). Наш viewer читает
   **только KMZ** и от structure.json не зависит — **доп. раунд не нужен**.
   Чеклист `CHANGELOG_052 §7 «для команды viewer»` к нам неприменим (он про
   гипотетический JSON-viewer). **Просьба:** подтвердите KMZ-only как
   единственную контрактную поверхность — закрепим в контракте окончательно.
5. **Реструктуризация каталогов (`viewer/`/`worker/`).** Согласны. Делаем
   отдельным PR `shared/repo-layout` (только `git mv`, без правок кода)
   **нашими руками**, но **после** связки формата в main (S2) и **до**
   разделения ролей (S4) — порядок продиктован приоритетом владельца
   «сначала формат». При переезде проверим scope регистрации `sw.js` и
   base-path GitHub Pages, отметим в описании PR. Вы синхронно поправите
   свои импорты/ссылки, как и обещали в §5 вашего письма.
