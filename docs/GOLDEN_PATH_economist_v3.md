# Золотой путь обработки данных

Документ для экономиста. Описывает **рекомендуемую последовательность**
запуска скриптов для получения структурированных данных по объекту
недвижимости и построения визуальной карты — от исходных документов до
интерактивного HTML-графа и KMZ для Google Earth.

> Предполагается, что у вас уже есть **VS Code** + активированный `venv`
> для проекта (см. `INSTRUCTION_economist.md`).

---

## 1. Установка зависимостей

В терминале VS Code с активным `venv` (один раз на машину):

```powershell
pip install Pillow piexif pymupdf openpyxl
```

Для конвертации `.doc`/`.docx` дополнительно требуется один из:
- **LibreOffice** (бесплатно) — https://www.libreoffice.org/, либо
- **MS Word** + `pip install comtypes` (только Windows).

Если ни одного из этих пакетов нет, скрипт продолжит работать — PDF
обрабатываются штатно, а `.doc/.docx` пропускаются с подсказкой.

  `Сооружения`, `Помещения`, `ОНЗ` — создаются автоматически, только

## 3. Шаги обработки

### Шаг 0. Создать новую болванку проекта

```powershell
python pirushin_sosn_rocha_07_init_project_v1.py
```
→ **меню → 1. Создание структуры** → корневая папка (Enter — `D:\ОБЪЕКТЫ`)
→ имя проекта → дерево создаётся, путь копируется в буфер обмена.

### Шаг 1. ЕГРН-выписки (PDF → JSON)

**Скрипт:** `pirushin_sosn_rocha_02_parsing_egrn_vipiska_v*.py`
**Вход:** PDF в `Выписки_PDF\ЕГРН\` (рядом можно класть парный XML —
он имеет приоритет при сводке).
**Выход:** `json\egrn_<TS>.json`

### Шаг 2. Свидетельства о собственности (PDF → JSON)

**Скрипт:** `pirushin_sosn_rocha_02_parsing_svidetelstva_nedvig_2000x.py`
**Вход:** PDF в `Выписки_PDF\Свидетельства_о_праве\`
**Выход:** `json\parsed__svidetelstva_<TS>.json`

### Шаг 3. KML/KMZ-разметка (KML → JSON)

**Скрипт:** `pirushin_sosn_rocha_01_parsing_kml_*`
**Вход:** KML/KMZ-файл с Placemark-ами (`Выписки_PDF\Прочее\` или
`kmz-kml\`).
**Выход:** `json\session_export_<TS>.json`

### Шаг 4. NSPD-обогащение (онлайн → JSON)

**Скрипт:** `pirushin_sosn_rocha_01_parsing_nspd_gemini_v5.py`
**Вход:** список КН + интернет.
**Выход:** `json\nspd_objects_<TS>.json`, кэш ответов кладётся в
`_data\nspd_cache\`.

### Шаг 5. Сведение источников

**Скрипт:** `pirushin_sosn_rocha_03_enrich_v14.py`

```powershell
python 03_enrich_v14.py "D:\ОБЪЕКТЫ\<проект>\json\*"
```

Или интерактивно — по одному файлу подряд, Enter для завершения.

Скрипт сам считает этажность зданий из помещений. В отчёте появится
строка вида:
```
[+] Этажность обогащена по помещениям: ОКС=3
```
**Выход:** `json\enriched_<TS>.json`

### Шаг 6. ОСВ → структура

**Скрипт:** `pirushin_sosn_rocha_052_make_structure_v1.py`
**Вход:** `ОСВ\osv.xlsx`, файлы JSON в `json\`.
**Запуск:** `python pirushin_sosn_rocha_052_make_structure_v1.py`,
интерактивные ответы:
1. Путь к `.xlsx` ОСВ.
2. (Enter) или путь к предыдущему `structure_*.json` как базе.
3. Путь к каталогу с JSON-источниками (`json\`).
4. Корневая папка с фотографиями (`Фотографии\`) — Enter, чтобы
   пропустить.
5. Пройти Фазу А (BU по адресам) и Фазу Б (BU по эвристике).

**Выход:** `_data\structure.json` — итоговая иерархия предприятия
(BU → объекты → уровни → оборудование) с высотами по русской системе.

#### Как организовать фотографии

Скрипт распознаёт **кадастровые номера в именах папок** по правилу:
- двоеточие `:` заменено на подчёркивание `_`;
- слэш `/` (для частей ЗУ/ОКС) заменён на двойное подчёркивание `__`.

Примеры:
- `Фотографии\Недвижимость\Строения\61_44_0050706_31\Фасад_север\IMG.jpg`
  → КН `61:44:0050706:31`.
- `Фотографии\Оборудование\01_1192\Торшер.jpg` → инв-номер `01_1192`.

Сопоставление работает также по:
- адресу (если имя папки содержит ≥ 2 токенов из адреса объекта);
- slug имени бизнес-единицы.

Не нужно строго следовать шаблону имени папки — главное, чтобы где-то в
цепочке родительских папок встречался КН-маска или инв-номер.

### Шаг 7. Конвертация документов в JPG и синхронизация фото-дерева

```powershell
python pirushin_sosn_rocha_07_init_project_v1.py
```
→ **меню → 2. Конвертация PDF → JPG** → Enter (берёт последний проект).

Скрипт:
- рекурсивно обходит `Выписки_PDF\` и `doc\`,
- конвертирует `.pdf`, `.doc`, `.docx` в JPG → `Документы_JPG\<подпапка>\`,
- даёт машиночитаемые имена: `egrn_<КН>_d<ГГГГММДД>_p<NN>.jpg`,
  `svid_<КН>_…`, `tehpasp_<КН>_…`, `tehplan_<КН>_…`, `doc_<КН>_…`,
  `egrul_inn<10цифр>_…`, `egrip_innfl<12цифр>_…`,
- пишет EXIF: GPS центра объекта (если известен) + UserComment-JSON
  (КН, ИНН, ОГРН, дата документа, страница из общего числа, высота
  объекта, отметка AMSL),
- **идемпотентен**: повторный запуск пропускает уже сконвертированные
  страницы; задвоенные PDF (тот же документ в разных подпапках)
  отбрасываются целиком, без разворачивания по страницам.

После конвертации скрипт **синхронизирует фото-дерево**: на основании
обнаруженных источников (XML-выписки, `structure.json`, любые JSON в
`json\`, EXIF документов, имена JPG, KML/KMZ) создаёт подпапки
`Фотографии\Недвижимость\<категория>\<КН>\` и при необходимости `План\`.

Параллельно скрипт пишет/обновляет **паспорт объекта**
`json\objects\<КН с _>.json` — единый агрегат по объекту: тип, адрес,
площадь, кадастровая стоимость, координаты центра, высоты, перечень
JPG-документов.

### Шаг 7а (опц.). Сортировка хаотичных фотопапок

Если у вас есть исторический «дамп» — папки с фото, документами и
выписками в произвольных подкаталогах (типичная картина: адрес-папка →
`Документы\…\<КН>\*.pdf`, `Фото\Состояние внутри\Вид от окна\…`,
вперемешку с опечатками и сокращениями вроде «ЗУ», «зу»,
«земучасток», «подавал», «зучесток») — положите всё дерево в папку
`Не_распределено\` **в корне болванки** (рядом с `ОСВ\`, `Фотографии\`,
`Выписки_PDF\`, `json\`), а затем запустите:

```powershell
python pirushin_sosn_rocha_07_init_project_v1.py
```
→ **меню → 3. Сортировка папки Не_распределено/** → Enter (последний проект).

Скрипт:
- разносит **любые типы файлов** по корневым папкам:
  - `*.pdf` → `Выписки_PDF\<подпапка по виду>\` (ЕГРН / ЕГРЮЛ-ЕГРИП /
    Свидетельства_о_праве / Технические_паспорта / Техпланы / Прочее)
  - `*.xml` → та же подпапка, что и у парного PDF (или ЕГРН)
  - `*.doc, *.docx` → `doc\`
  - `*.xls, *.xlsx` → `xlsx\`
  - `*.db` → `db\`
  - `*.html, *.htm` → `html\`
  - `*.json` → `json\`
  - `*.kml, *.kmz` → `kmz-kml\`
  - `*.txt, *.md` → `txt-md\`
  - `*.sig, *.asc, *.p7s` (ЭП-подписи) → `Выписки_PDF\Прочее\`
  - `*.jpg, *.jpeg` → `Фотографии\Недвижимость\<категория>\<КН>\<sem>\`
- найдёт **КН** в путях (с двоеточиями, подчёркиваниями, и «плоский»
  12-15 цифр подряд: `61440040713308` → `61:44:0040713:308`);
- определит **тип объекта** по словам в путях (с опечатками):
  *квартира*, *жилой дом*, *здание*, *сооружение*, *ЗУ/з/у/земучасток/
  зучесток*, *ОНЗ/незавершённое*;
- нормализует **семантические подпапки**: `Вход_в_объект`,
  `Общий_вид_адреса`, `Прилегающая_территория`, `Состояние_внутри`,
  `Состояние_отделки`, `Состояние_объекта`, `Подвал`, `Чердак`,
  `Вид_от_окна`, `План_эвакуации`, `Планы`, `Экспликация`,
  `Кухня`, `Санузел`, `Фасад`, `Литер_<буква>`, `Кв_<номер>`,
  `Этаж_<N>`, `Аналоги`, `Окружение_квартала`;
- проиндексирует **любые `egrn_*.json` в дампе** (таблицы
  `building_objects`, `land_objects`): сразу получает `{КН: адрес,
  тип_объекта}`, без отдельного шага парсинга выписок;
- построит индекс **«адресная папка → КН»** за один проход: если в
  дампе встречается подпапка с КН в названии
  (`Ростов-на-Дону_Суворова_52/Документы/61440040713308_квартира_…/`),
  её ближайший НЕ-контейнерный предок становится «адресной папкой»
  этого КН; все фото внутри неё привязываются автоматически. Из имени
  подпапки также извлекается **подсказка категории** (`…_квартира_…` →
  room, `…_жилой_дом_…` → building, `…_зу_…` → land);
- для каждого фото будет искать КН в братских PDF/XML **во всех
  родительских папках** (не только в той же папке) — это позволяет
  привязать фото `БСадовая_111/Состояние объекта/IMG.jpg` к КН из
  PDF, лежащего тремя уровнями глубже в `БСадовая_111/Документы/<КН>/`;
- сопоставит адреса любой родительской папки с индексом дампа
  и `json\objects\` по ≥ 2 общим токенам адреса;
- сохранит план в `json\photo_migration_plan.json`, покажет таблицу
  «откуда → куда» и спросит подтверждение **перед** перемещением.

**При подтверждении** файлы переносятся (`move`), сохраняя `mtime/atime`.
Для JPG дополнительно дописывается **EXIF**: `ImageDescription` и
`UserComment` (JSON с КН, категорией и семантикой). При этом
**оригинальные EXIF не затрагиваются**: `DateTime`, `DateTimeOriginal`,
`DateTimeDigitized`, GPS-координаты и высота остаются как были.

**Полная исходная иерархия** меток сохраняется (например:
`Фотографии\Недвижимость\Помещения\61_44_0040713_308\Состояние_внутри\Вид_от_окна\IMG.jpg`).

**Если по одному адресу несколько КН** (здание + ЗУ + помещения) и в
имени файла нет уточняющего КН — фото остаются на месте, в логе
печатается список кандидатов; вы сами разнесёте их вручную.

**Если КН так и не определён** (нет КН в путях, EXIF или соседних PDF,
и нет адреса в `json\objects\`) — файлы остаются в корневой папке
`Не_распределено\` как есть.

### Шаг 8. Построение графа (с оборудованием и уровнями)

**Скрипт:** `pirushin_sosn_rocha_04_nspd_graph_v14.py`

**Два варианта запуска:**

**A. Только юридический граф:**
```powershell
python 04_nspd_graph_v14.py "D:\…\json\enriched_<TS>.json"
```
На графе: объекты, права, обременения, бенефициары, бизнес-единицы,
учредители.

**B. Полный граф с оборудованием и уровнями (рекомендуется):**
```powershell
python 04_nspd_graph_v14.py "D:\…\json\enriched_<TS>.json" "D:\…\_data\structure.json"
```
Или интерактивно: скрипт сначала спросит путь к enriched, затем путь к
structure (Enter — пропустить).

На графе **дополнительно** появятся:
- узлы уровней (этажи, подвалы, антресоли) — жёлтые квадраты внутри
  здания;
- узлы оборудования (ОС из ОСВ) — серые точки, привязанные к BU,
  уровням или зданиям;
- рёбра «уровень в строении», «на уровне», «ОС бизнес-единицы».

В правой панели фильтров появятся переключатели **«Уровни (этажи)»** и
**«Оборудование (ОС)»** — можно временно скрыть тысячи мелких узлов
оборудования, чтобы рассмотреть юридический скелет.

**Выход:** `html\graph.html`. Открыть двойным кликом в Chrome / Edge /
Firefox. Нужен интернет для загрузки vis-network.

## 4. Что на графе означает каждый узел и связь

### Узлы

| Цвет / форма | Тип | Что это |
|---|---|---|
| 🟩 ромб зелёный | Земельный участок | объект из ЕГРН |
| 🟧 квадрат оранжевый | Здание | объект из ЕГРН |
| 🟦 кружок голубой | Помещение | объект из ЕГРН |
| 🟪 квадрат фиолетовый | Сооружение | объект из ЕГРН |
| 🟨 квадрат жёлтый | Уровень (этаж/подвал) | из `_data\structure.json`, рассчитан по этажности |
| ⚪ кружок серый маленький | Оборудование (ОС) | из ОСВ через скрипт 052 |
| 🟥 квадрат красный | Бизнес-единица | из enrich (адрес + ОГРН) + 052 (поглощённые) |
| 🔴 кружок красный | Бенефициар (юр.лицо) | из ЕГРЮЛ |
| 🩷 кружок розовый | Бенефициар (физ.лицо) | из выписок |
| 🔵 кружок синий | Право | из ЕГРН (собственность, аренда, …) |
| 🔷 треугольник тёмный | Обременение | из ЕГРН (залог, культурное наследие, …) |

### Рёбра

| Цвет | Связь |
|---|---|
| зелёная сплошная | «На участке расположен» (ЗУ → здание) |
| оранжевая сплошная | «Помещение в здании» (здание → помещение) |
| золотистая сплошная | «Уровень в строении» (здание → этаж) |
| серая сплошная | «На уровне» (этаж → оборудование) |
| серая пунктирная | «На объекте» (здание/ЗУ → оборудование) |
| красная пунктирная | «ОС бизнес-единицы» (BU → оборудование) |
| синяя сплошная | «Имеет право» (объект → право) |
| красная сплошная | «Правообладатель» / «В пользу» (право → бенефициар) |
| красная сплошная | «Хозяйственная деятельность» (объект → BU), «Ведёт» (BU → бенефициар) |
| фиолетовая | «Учредитель X%» (бенефициар → бенефициар) |

## 5. Типовые сценарии

### Первичная сборка проекта

1. Шаг 0 (`07_init_project`, меню 1): болванка готова.
2. Шаги 1–4: собираем JSON-источники в `json\`.
3. Шаг 5 (`03_enrich_v14`): получаем `json\enriched_*.json`.
4. Шаг 6 (`052_make_structure`): получаем `_data\structure.json`.
5. Шаг 7 (`07_init_project`, меню 2): конвертация документов в JPG +
   синхронизация фото-дерева + паспорта объектов.
6. Шаг 8B (`04_nspd_graph` + structure): полный граф в `html\`.
7. Шаг 9 (`08_build_kmz`): KMZ-карта в `kmz-kml\`.

### Свежая ОСВ через квартал

1. Заменить `ОСВ\osv.xlsx` на новую выгрузку.
2. Запустить Шаг 6 со старым `_data\structure.json` как базой —
   ваши правки имён BU сохранятся.
3. Перезапустить Шаги 7, 8B, 9 — данные обновятся.

### Добавилась новая выписка ЕГРН

1. Положить PDF в `Выписки_PDF\ЕГРН\` (рядом — XML, если есть).
2. Шаг 1 для парсинга в JSON.
3. Шаг 5 (повтор) — этажность зданий обновится автоматически.
4. Шаг 7 — конвертация новой выписки в JPG + автоматическая подпапка
   `Фотографии\Недвижимость\<категория>\<КН>\`.
5. Шаги 8B, 9 — теперь на графе и KMZ появятся новые узлы.

### Изменили адрес здания в NSPD

1. Шаг 4 заново подтянул новый адрес.
2. Шаги 5, 6 — структура обновится, но ID бизнес-единиц остаются
   прежними (якорь — КН, не адрес). Ваши правки имён BU сохранятся.
3. Шаги 7, 8, 9 — артефакты пересоберутся.

## 6. Проверка устойчивости

Запустите всю цепочку **дважды подряд** с одним и тем же набором входов
и сравните:
- размер `json\enriched_*.json` — должен совпасть бит в бит (если
  ничего внешне не изменилось);
- набор `id` в `structure.equipment[]` / `business_units[]` —
  идентичен;
- HTML-граф — одинаковое количество узлов и рёбер каждого типа;
- `kmz-kml\project.kmz` — одинаковое содержимое (сравнить размер
  списка JPG/полигонов внутри zip).

Если что-то разошлось — баг. Сообщите разработчику и приложите оба
output-файла.

## 8. Высоты и подготовка к KMZ

Скрипт 052 рассчитывает высоты по **русской системе**: этаж = 3 м,
подвал = −3, полуподвал = −2, цокольный = −2, антресоль = +2 над своим
этажом, мансарда = +высота_здания, технический этаж сверху = +(N+1)×3.

### Что вы получите в `_data\structure.json` и `json\objects\<КН>.json`

| Объект | Поле | Содержит |
|---|---|---|
| Здание / Сооружение / ОНЗ | `height_m` | надземная высота, м |
|  | `depth_m` | глубина подземной части, м (если есть) |
|  | `_geometry.extrude` | `true` — здание готово к 3D-extrude |
|  | `_geometry.top_z_meters` | верхняя отметка, м |
|  | `_geometry.bottom_z_meters` | нижняя отметка, м |
| Уровень (этаж/подвал) | `z_meters` | z пола над «уровнем 0» |
|  | `top_z_meters` | z потолка |
| Оборудование | `z_meters` | z пола его уровня |
| Фото (EXIF) | `exif.gps_lat/lon/alt/bearing/datetime` | сырые EXIF-данные |
|  | `z_meters` + `z_source` | вычисленная высота |
|  | `is_unassigned` | `true` для фото из `Не_распределено` |

### Папка `Не_распределено\` (в корне болванки)

Сюда кладут (а) фотографии, которые **не должны участвовать** в 3D —
например, общий вид территории без привязки к конкретному зданию,
который иначе вытянул бы 3D-сцену вверх по случайной EXIF-высоте
(GPS = 100 м над уровнем моря); (б) любые «хаотичные» материалы
пользователя (PDF/XML-выписки, DOC/XLS-договоры, db/html/JSON, KML)
до запуска меню 3. У фото в этой папке:
- читается EXIF (координаты, дата);
- но `z_meters` остаётся пустым (`is_unassigned: true`).

После запуска меню 3 (см. Шаг 7а) большинство файлов разносится по
структуре проекта; то, что не удалось привязать, остаётся в
`Не_распределено\` как есть.

### KMZ-экспорт (Шаг 9)

JSON содержит достаточно данных для построения KMZ:
- **Здания → 3D-полигоны:** bottom_z..top_z по геометрии участка.
- **Уровни → горизонтальные сечения** на z_meters.
- **Оборудование → точки** на высоте z_meters.
- **Фотографии → метки** в координатах EXIF GPS и высоте z_meters.

## 9. Частые проблемы

| Симптом | Причина | Решение |
|---|---|---|
| «z_meters=null у всех фото» | Pillow не установлен, или нет EXIF/привязки к уровню | `pip install Pillow` |
| «Все фото на одной высоте над морем» | Используется EXIF altitude (метры над уровнем моря) | Раскладывайте фото по `Фотографии\Недвижимость\Строения\<КН>\Этаж_N\…` — z будет браться из уровня |
| «Здание плоское в 3D» | У ОКС не вычислилась этажность (`height_m=null`) | Проверить выписку ЕГРН — должен быть атрибут «Количество этажей» |
| «Этажность пустая» в графе | У ОКС нет ни «Количество этажей», ни помещений с указанием этажа | Обновить выписку ЕГРН или дополнить вручную |
| «На графе две BU с одинаковым именем» | Cross-link не нашёл совпадения КН | Проверить, что `anchor_cadastral` BU 052 совпадает с «Объект (КН)» BU enrich |
| «Слишком много узлов оборудования, граф тормозит» | На большом проекте 700+ ОС | В фильтрах справа снять галочку «Оборудование (ОС)» — узлы скроются, скелет графа останется |
| «Шаг 8B не нашёл structure» | Опечатка в пути | Граф запустится только с enriched (без узлов eq/level), это не ошибка |
| «Этажность обогатилась некорректно» | В наименовании помещения числа похожи на этажи | Открыть `json\enriched_*.json`, найти `_floors_inferred_sources` — это список КН, по которым считалось; проверить вручную |
| «.doc/.docx не конвертируются» | Не найден ни LibreOffice, ни MS Word | Поставьте LibreOffice (https://www.libreoffice.org/) или, если на Windows есть Word — `pip install comtypes` |
| «`structure.json пока не создан`» | Шаг 6 ещё не запускался | Это нормально для свежей болванки; запустите Шаг 6 и повторите Шаг 7 — JPG получат GPS |
| «Папка `Фотографии\Недвижимость\<КН>\` не создалась» | Тип объекта ещё не определён | Положите XML-выписку или прогоните Шаги 4–6, чтобы тип был известен; повторите Шаг 7 |
| «`[skip] file.pdf (дубль, уже сконвертирован)`» | Идемпотентность: тот же документ уже обработан | Чтобы переконвертировать — удалите старые JPG этого документа в `Документы_JPG\` |
