## Суть

Stage 6 ЭТП-экспортёра автоматически сводит `kind:"photo"` JPG в `extras.advantages[]` («Комплексная фотофиксация: Фасад, Кровля…»). Но информация о состоянии каждого конкретного фото — комментарий экономиста («трещина по фасаду в правом углу», «новая кровля 2024») — сейчас никуда не идёт.

Предлагаю аддитивный bump EXIF UserComment v1.1 → v1.2:

| Ключ | Тип | Где |
|---|---|---|
| `note` | string \| null | новое опциональное поле в payload `kind:"photo"` |

Бакэнд-следствия (после ack):

1. Парсер `parser/scripts/pirushin_sosn_rocha_07_init_project_v*.py` — опционально записывает `note` (если экономист его задал).
2. Stage 6 `etl_exif.py` агрегирует `note`-поля по `cad_number` → `object_etp_profile.extras.notes` (joined `«; »`).
3. `docs/EXIF_USERCOMMENT_SCHEMA.md` v1.1 → v1.2 (только аддитивно).

## Контрактные инварианты (по аналогии с v1.0 → v1.1)

- Аддитивно. `note` опционален. Старые JPG без поля по-прежнему валидны.
- Парсер v1.2 продолжает читать v1.1 JPG (note=None).
- Viewer v1.1 ignore'ит неизвестный ключ — не падает.
- Поле не часть wire-формата KMZ, не часть `CONTRACT_KMZ.md` инвариантов.

## Что прошу обсудить (5 вопросов)

### 1. Имя поля

- **(a) `note`** — единое поле, заметка экономиста.
- (b) `notes` — массив строк (несколько заметок на одну JPG).
- (c) Что-то структурированное (`{text, author, ts}`).

**Parser-рекомендация:** (a). Заметка обычно одна на фото; если хочется нескольких — экономист пишет через `; ` сам, парсер join'ит как есть. Структурированную форму вводим только когда появится UX-кейс (трекинг автора/времени правки).

### 2. Где экономист вводит note

- **(a) Расширение существующего viewer/admin-etp-profile.html** — добавить опциональную форму «Заметки к фото объекта» с массивом «cad + note». При генерации YAML → ?
- (b) Отдельный admin/etp-photo-note/&lt;cad&gt; UI viewer-team.
- (c) Не делаем UI в v1, экономист правит EXIF вручную через сторонний инструмент.

**Parser-рекомендация:** (a) **с оговоркой**. Через YAML мы не можем передать заметки обратно в EXIF (это про JPG, не БД-профиль). Поэтому viewer-UI должен либо:
- (a1) сразу писать `note` в EXIF JPG-файла (требует backend-canal для записи в файл, что противоречит «viewer = статика GitHub Pages»);
- (a2) генерировать дополнительный YAML с `extras.notes` напрямую (минуя EXIF) — тогда поле живёт только в БД-профиле, фото не модифицируется.

(a2) самое лёгкое: вводим в admin UI новое поле `extras.notes_per_photo[]` или просто текстовое `extras.notes`, экономист пишет туда; парсер пишет в БД; «комплексная фотофиксация» по-прежнему собирается из категорий JPG.

### 3. Куда note попадает в БД

- **(a) `extras.notes`** (одно текстовое поле всего объекта; сейчас уже существует) — добавляем туда per-фото заметки joined «; ».
- (b) Новое поле `extras.photo_notes[]` (массив строк).
- (c) Новая колонка `object_etp_photo_notes` (отдельная таблица).

**Parser-рекомендация:** (a) — переиспользуем существующее `extras.notes`. Минимум миграционных изменений. Если в будущем нужна гранулярность — поднимем bump схемы БД отдельным циклом.

### 4. Что делает Stage 6 EXIF в режиме v1.2

- Читает все JPG `kind:"photo"` для cad.
- Собирает категории → строка «Комплексная фотофиксация…» (как сейчас).
- **Новое v1.2:** собирает `note`-поля → добавляет в `extras.notes` joined «; » (gap-fill, не перезаписывает существующее значение).

Поведение остаётся идемпотентным (повторный прогон не меняет уже записанное).
