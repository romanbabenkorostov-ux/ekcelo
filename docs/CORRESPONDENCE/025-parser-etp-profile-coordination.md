# 025 — Согласование ЭТП-профиля (object_etp_profile / lots / lot_items) и UI его редактирования

- **From:** parser
- **To:** viewer
- **Date:** 2026-05-27
- **Re:** PR #41 (SPEC `docs/etp_export/SPEC_etp_export.md`); PR #48 (ADR-001 `obsidian/Decisions/ADR-001-etp-profile-extension.md`)
- **Status:** open

## Суть

PR #41 ввёл SPEC экспорта объектов сюрвея в три ЭТП (`torgi.gov.ru`, `roseltorg.ru`, `sberbank-ast.ru`). PR #48 зафиксировал ADR-001: новые таблицы `object_etp_profile`, `lots`, `lot_items` в основной БД как явно не-ЕГРН слой с полями `source` ∈ {`osv`, `exif`, `manual`, `nspd`, `llm`} и `confidence`. Принцип переформулирован: «БД = слепок ЕГРН + ЭТП-профиль».

Следующий шаг — PR с миграцией DDL. Перед ним нужно согласовать с viewer-team пять вопросов по интеграции. Контракт KMZ 2.12.0 не затрагивается (§3 UI/UX-домен), wire-формат прежний.

## Вопросы (нужен ack по каждому)

### 1. UI редактирования профиля
Кто реализует редактор полей `object_etp_profile` (`location_extra`, `building_extra`, `layout`, `legal_extra`, `risks`, `extras`)?

- (a) viewer добавляет inline-форму редактирования прямо в карточке объекта;
- (b) отдельный мини-UI (страница `admin/etp-profile/<cad_number>`);
- (c) только bulk-import из XLSX/YAML, без UI редактирования полей.

**Parser-рекомендация:** (b) для MVP — изолирует ЭТП-логику от основного viewer, можно скрывать токеном (как `viewer/admin-encode.html`).

### 2. Отображение `source` / `confidence`
Как отрисовывать поле с `source='llm'` и `confidence < 0.7`?

- (a) бейдж рядом с полем («по данным внешних источников», цветовая индикация);
- (b) тултип с расшифровкой источника и confidence;
- (c) приглушённый цвет шрифта + значок;
- (d) скрывать пока не подтверждено экономистом (`source='manual'`).

**Parser-рекомендация:** (a) + (b) — бейдж видимый, тултип с деталями. Скрытие (d) ломает прозрачность.

### 3. `lots` / `lot_items` в viewer
Должен ли viewer показывать группировку «лот = группа КН» — например, новая вкладка «Лоты» с агрегацией?

- Если **да**: список лотов на главной → переход в состав лота с подсветкой объектов на карте графа.
- Если **нет**: это только бэк-функция для экспортёра (генерация артефактов лота в `out/etp/<lot_id>/`).

Parser-рекомендация отсутствует — это viewer-домен (§3 UI/UX).

### 4. Источник правды для совместных полей
Если объект имеет `address` в `objects` (из ЕГРН) и `location_extra.landmark` в `object_etp_profile` — как показывать их в карточке?

- (a) смешанно одним блоком (как читаемый текст);
- (b) раздельно с подписями «ЕГРН» / «ЭТП-профиль» (двухколонный layout или две секции).

**Parser-рекомендация:** (b) — происхождение каждого поля прозрачно; согласовано с §1.3 SPEC «маркировка источника».

### 5. Сроки и приоритет
Когда viewer-team может взять задачу на UI ЭТП-профиля?

- Параллельно с миграцией парсера — нужен контракт на read-only fixture JSON, на котором viewer работает до готовности backend writes.
- Или после — viewer стартует, когда parser-side готов писать в `object_etp_profile`.

Если параллельно: parser-A может отдать fixture `tests/fixtures/etp/object_etp_profile_sample.json` (схема согласно ADR-001 §Решение) в течение 1–2 дней после ack.
