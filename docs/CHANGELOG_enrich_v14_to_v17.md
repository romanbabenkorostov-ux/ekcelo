# Обогатитель `03_enrich_*.py` — что менялось от v14 к v17

Этот документ — для разработчиков, поддерживающих пайплайн NSPD → 022 → enricher.
Описаны только содержательные изменения и причины. Внутренний рефакторинг без эффекта не упоминается.

---

## Восстановленные вводные

По переписке и присланным файлам контур задачи выглядит так:

- Объекты недвижимости в РФ (Крым, Москва и др.). Источник истины — НСПД, выписки ЕГРН, kadbase/rosreestr/egrp365 (через ручной парсер `022`).
- Юрлица / ИП / ФЛ — через сторонний сервис, отдающий JSON (`LEGAL_*`, `PERSON_*`), и через PDF-выписки ФНС (ЕГРЮЛ / ЕГРИП).
- Цель пайплайна — собрать в одном `enriched.json` объекты недвижимости + бенефициаров + цепочки учредительства + бизнес-единицы (по совпадению юр. адреса с адресом ОКС).
- Среда: Windows 10, Python 3.13, venv в `E:\Code\!util\сoder_exif\backend`, скрипты в `E:\Code\!util\parsing_nspd`.
- Идемпотентность важна: повторный запуск с теми же входами в ту же папку — не порождает новых файлов и не теряет ранее принятых решений сопоставления.
- В консоли — короткий и информативный лог: сразу видно, что подключилось, какие бенефициары и их роли.

## v14 → v15 — нормализация атрибутов NSPD и фикс этажности

**Причина.** В session_export 25.0 из `01_parsing_nspd_v7` поля приходят в произвольных русскоязычных названиях и в виде строк («7 022 кв. м», «2019 г.», «-»). Это ломает сравнения и числовую агрегацию. Параллельно в v14 уже была функция `enrich_floors_from_premises`, но она опиралась на исторические имена ключей и **молча затирала** этажность EGRN-объектов (бага в v14, замаскированная отсутствием EGRN-источников в текущих тестах).

**Что сделано:**

- `AREA_RE` — поддержка разрядных пробелов (`\u00a0`, обычный пробел): `«7 022 кв. м»` → `7022`.
- `extract_area` — удаление пробелов перед `float()`.
- Новый словарь `NSPD_FIELD_ALIASES` + функция `_nspd_normalize_attrs`. Канонические имена выровнены с `EGRN_BUILDING_FIELDS` / `KADBASE_FIELD_ALIASES`:
  - `Дата присвоения` → `Дата постановки на учёт`
  - `Площадь` / `Площадь общая` / `Площадь уточненная` → `Площадь, кв.м`
  - `Количество этажей (в том числе подземных)` → `Количество этажей всего`
  - `Завершение строительства` → `Год завершения строительства`
- Числовые поля (`Количество этажей всего`, `Год завершения строительства` и т.д.) приводятся к `int`.
- Строковые пустышки (`-`, `—`, `–`, `«не указано»`, `«нет данных»`) выбрасываются.
- `enrich_floors_from_premises` ищет существующую этажность под **всеми** возможными именами ключа (`Количество этажей`, `Количество этажей всего`, `Этажей надземных`, …) и пишет результат в EGRN-канонические имена. Известное значение этажности больше не затирается.

**Результат на присланном session_export:** 101/101 объектов с площадью получили числовое значение (было 0/101).

## v15 → v16 — парсинг PDF-выписок ФНС напрямую

**Причина.** Для ЮЛ есть JSON от стороннего сервиса. Для ИП — только PDF из ФНС. Чтобы не плодить промежуточные конвертеры, enricher теперь ест PDF сам.

**Что сделано:**

- Опциональный импорт `pdfplumber` (если нет — PDF-источники отклоняются с подсказкой `pip install pdfplumber`).
- `load_file` для `.pdf` вызывает встроенный `parse_fns_pdf`.
- Парсер построен на `pdfplumber.page.extract_tables()` — выписка ФНС имеет стабильную структуру `[№, поле, значение]`. Заголовки секций в табличный поток не попадают, поэтому маршрутизация по содержимому полей:
  - Маркер открытия блока учредителя/директора — поле `«ГРН и дата внесения в ЕГРЮЛ сведений о данном лице»`.
  - Глобальные «закрыватели» блока — поля типа `«Размер (в рублях)»`, `«ИНН юридического лица»`, `«Код и наименование вида деятельности»` и т.п.
  - Различение директор / учредитель — по наличию `«Должность»` либо `«Размер доли»` в собранном блоке.
- Защита от ложноположительного ОГРН: первое вхождение поля в верхнем уровне фиксируется через `seen_global`, повторные (например, для держателя реестра АО без маркера секции) игнорируются.
- ЕГРИП — отдельный путь, `entity.type = "ip"`, без учредителей/директоров/УК.
- `detect_file_kind` распознаёт `egrul_pdf` / `egrip_pdf` через поле `_source` в результате `parse_fns_pdf`.
- Новый `kind = "ip_entity"`, отдельный `parse_ip_entity`, отдельная ветка в `main`.
- `beneficiary_display_name` различает `_kind = "ip"` и формирует подпись `«ИП ФИО»`.

**Результат:** все 4 присланных PDF (3 ЕГРЮЛ + 1 ЕГРИП) парсятся корректно — ОГРН, ИНН, КПП, наименование, адрес, УК, ОКВЭДы, директор (ФИО+ИНН+должность), учредители с долями.

## v16 → v17 — PERSON, залоги долей, держатель реестра АО, идемпотентность, лог

Эта версия закрывает оставшиеся пробелы в полноте бенефициаров и аккуратности консольного вывода.

### Парсинг PERSON_*.json

Формат:

```json
{
  "entity": {"type": "person", "inn": ..., "fio": ..., "ip_list": []},
  "raw_data": {
    "Руковод": [<ЮЛ, где это лицо руководитель>],
    "Учред":   [<ЮЛ, где это лицо учредитель>]
  }
}
```

Что добавлено:

- Новый `kind = "person_entity"` и функция `parse_person_entity`.
- В `attrs` бенефициара ФЛ — две группы: «Является руководителем (организации)» и «Является учредителем (организации)», каждая с ОГРН/ИНН/Наим/Статус/Дата ликвидации/Юр.адрес/ОКВЭД.
- `founders_links` нового вида `kind = "person_to_legal"` (семантика инвертирована относительно `legal`: в этой ссылке founder — сам носитель PERSON, child — ЮЛ из ссылки).
- `build_founder_chains` дополнен веткой `person_to_legal`. Для упомянутых дочерних ЮЛ, которых нет среди загруженных бенефициаров, создаются узлы-заглушки `_kind = "legal_text"` с пометкой источника.

### Обременения долей в УК (залог)

Эта информация имеет ту же ценность, что и обременения по ЕГРН, поэтому извлекается во всех путях:

1. **Из LEGAL JSON** — `raw_data.Учред.РосОрг[i].Обрем = [{Тип, Залогодерж: {ОГРН, ИНН, НаимПолн}, Нотариал: {Номер, Дата}, НотариусФИО?}]`. Складывается в `attrs["Обременения долей в УК"]` и помечается на учредителе флагом `has_pledge`.
2. **Из PDF ЕГРЮЛ** — добавлена sub-state-машина внутри парсера блока учредителя:
   - `(None, "Сведения об обременении", "")` → открыть залог;
   - `(n, "Сведения о договоре залога", v)` → распарсить «Номер договора залога / Дата договора залога» с поддержкой переноса строки в значении (важно: пример `77/660-Н/77-2025-\n2-2145` — раньше обрывался на `\n`, теперь склеивается);
   - `(n, "Сведения о нотариусе ... фамилия имя отчество", v)` → ФИО нотариуса;
   - `(None, "Сведения о залогодержателе ... юридическом лице", "")` → флаг `pledge_holder_expected`. **Следующий** маркер «ГРН … о данном лице» открывает не нового учредителя, а holder-блок (важная развязка — раньше залогодержатель ошибочно открывался как новый учредитель).
3. В консольный лог добавлена строка `[!] ЗАЛОГ ДОЛИ → <банк>  (ИНН ...)` с номером и датой договора.

### Держатель реестра акционеров (для АО)

В выписке этой секции нет маркера в табличном потоке. Решение — fallback через `pdf.extract_text()` всего документа и regex поиск секции `«Сведения о держателе реестра акционеров»`. Из неё вытаскиваются `ОГРН (13 цифр)`, `ИНН (10 цифр)`, `Полное наименование`. В консоль выводится строка `[АО] держатель реестра акционеров: ... (ИНН ...)`.

Для LEGAL JSON просто читается `raw_data.ДержРеестрАО` (если непустой).

### Расширение `parse_legal_entity`

- Директора теперь содержат `{ФИО, ИНН, Должность}` (а не только должность с подписью «физическое лицо»).
- Добавлены поля: «Уставный капитал, руб.», «Обременения долей в УК», «Держатель реестра акционеров».

### Идемпотентность вывода

- В начале запускается `print_banner()` с одностраничной справкой о поддерживаемых источниках и выходных файлах.
- `ask_output_dir()` запрашивает папку для результатов (Enter → текущая).
- `ask_show_cross_links()` — да/нет: показывать ли кросс-связи руководителей/учредителей в консоли (для PERSON-карточек: при `n` печатается только сводное число «руководитель в N ЮЛ», без перечисления).
- Выходные файлы переименованы в фиксированные:
  - `<output_dir>/enriched.json`
  - `<output_dir>/match_decisions.json`
- Повторный запуск в ту же папку — перезаписывает файлы. `match_decisions.json` подгружается до диалогов сопоставления, поэтому ранее принятые решения переиспользуются автоматически. Файлы с timestamp-суффиксом больше не плодятся.

### Расширенный консольный лог

После каждого `[+]` файла теперь печатается краткая «карточка»:

- Для ЮЛ — руководители (ФИО + ИНН + должность), учредители (наименование + ИНН + доля), для АО — держатель реестра, для каждого учредителя — обременения доли (банк + ИНН + номер/дата договора).
- Для ФЛ — список ЮЛ, где руководит / где учредитель, с пометкой `(ликв.)` для ликвидированных.
- Для ИП — только заголовок (атрибуты ИП и так минимальны).

## Точки внимания при дальнейшем сопровождении

### Схема `beneficiaries` в `enriched.json` — расхождение v14 ↔ v17

**Важно для интеграторов (052 `link_with_enriched`, 04, downstream-консумеры).**

| Версия enricher | Где живут реквизиты (ИНН/ОГРН/КПП/Полное наименование) |
|---|---|
| **v14** | внутри `ben["attrs"]` (вложенный словарь) |
| **v17** | на верхнем уровне записи `ben` (плоско) |

Это вызвало скрытый баг в `052_v2` (до v2_1): `link_with_enriched` искал ИНН/ОГРН/КПП на верхнем уровне `ben.get("ИНН")`, что для v14-выгрузок всегда давало `None` → `enterprise.inn/ogrn/kpp` оставались пустыми. Баг обнаружен parser-team A в посте 011 (PR #27) при тестировании v17-цепочки.

**Фикс в `052_v2_1` (fallback на `ben["attrs"]`)** — намеренный, покрывает оба формата:

```python
attrs = candidate.get("attrs") if isinstance(candidate.get("attrs"), dict) else candidate
enterprise["inn"]  = candidate.get("ИНН")  or attrs.get("ИНН")  or enterprise.get("inn")
```

При canonical v17 первый `candidate.get("ИНН")` срабатывает; при legacy v14 фолбэк через `attrs.get("ИНН")`. **Не удалять** этот fallback при будущей чистке кода.

### Защитный фильтр в `link_with_enriched` (052_v2_1)

С v17 в `beneficiaries` появились `_kind = "ip" | "person" | "legal_text"`. При поиске юр.лица по `name_short` фильтруем по `_kind` И по `Тип субъекта` — чтобы ИП/ФЛ с похожим ФИО не подменили enterprise.

## Известные хрупкости

1. **PDF-парсер залога** опирается на стабильную нумерацию строк выписки ФНС. Если ФНС поменяет состав/порядок полей в секции «Сведения об обременении» — sub-state-машина потребует подстройки. Тестовый якорь: выписка `ul-1149102128328` (Санаторий «Сосновая Роща»).
2. **Держатель реестра** через regex по `extract_text()` хрупок к изменению заголовка секции. Сейчас ищется подстрока `«Сведения о держателе реестра акционеров»`.
3. **PERSON-карточка** считает дочернюю компанию связью даже если она ликвидирована (`Статус = «Не действует»`). В цепочках это помечается, но не отбрасывается. Если нужен фильтр — править `parse_person_entity` (там сейчас уже отбрасываются ликвидированные дочерние при формировании `founders_links`, но не из `attrs`-списка).
4. **`enriched.json` идемпотентен по имени, не по содержимому** — `metadata.exported_at` обновляется при каждом запуске. Если нужна строгая контентная идемпотентность — заменить timestamp на хэш входных файлов.
5. **Кодировка консоли Windows.** Префиксы (`[Р]`, `[У]`, `[АО]`, `[!]`) — **кириллица в скобках**, не ASCII (опечатка ранней редакции CHANGELOG, отмечено parser-team A в посте 011). Выбраны вместо эмодзи, чтобы корректно отображаться на Windows-консоли при `chcp 65001` (UTF-8) и при `chcp 866` (CP866) без необходимости настройки терминала. Латинизация (`[D]`, `[F]`, `[RH]`, `[!]`) намеренно не делалась — кириллические аббревиатуры однозначно читаемы русскоязычным оператором. Если в будущем будет принято решение о латинизации (например, для CI-логов без локализации) — это разовая правка `_print_legal_beneficiaries` / `_print_person_links`.
