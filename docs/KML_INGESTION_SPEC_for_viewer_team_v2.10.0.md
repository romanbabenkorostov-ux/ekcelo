# Расширение спецификации KML для приёма в Ekcelo: v2.9.62 → v2.10.0

**Кому:** команда веб-приложения Ekcelo (карта/просмотрщик, Leaflet + JSZip + DOMParser, sw-cache WMS nspd.gov.ru).
**От:** команда парсера (Python 3.13+, Win10 RU), `pirushin_sosn_rocha_08_build_kmz_v2.py`.
**Дата:** 2026-05-18 · Версия контракта-предложения: **2.10.0** (надстройка над вашей v2.9.62) · Статус: запрос на расширение.

---

## Введение

Это письмо — ответ-предложение на вашу спецификацию `KML_INGESTION_SPEC_for_parser_team.md` v2.9.62. Мы внимательно её прочитали и принимаем целиком: семь обязательных пунктов из §0 (TL;DR), правила `<styleUrl>` с префиксами, формат `description` как пары `Ключ: значение; `, одна геометрия на Placemark, UTF-8 + CDATA для кириллицы, цветовая модель ABGR, дедуп по кад.№ и сигнатуре. Никаких возражений по существу контракта у нас нет.

Однако наша исходная модель данных (`structure.json` из `04_nspd_graph`) различает **шесть классов объектов, которых нет в текущем контракте v2.9.62**: помещения (квартиры, нежилые, машино-места), сооружения (линейные и точечные — газопроводы, ЛЭП, трансформаторные), ОНС (объекты незавершённого строительства), бизнес-единицы (филиалы, обособленные подразделения), оборудование (инвентарные объекты на балансе с привязкой к КН и этажу), бенефициары (юридические и физические лица из ЕГРЮЛ/ЕГРИП). Если положить их все в существующий `cad_oks_*`, мы потеряем семантику классификатора и сломаем дедупликацию.

Поэтому мы предлагаем **аддитивное** расширение контракта v2.10.0 — оно ничего не ломает в v2.9.62:

- Все существующие префиксы (`cad_zu_*`, `cad_oks_*`, `cad_exp_*`, `photoPin_*`) и три текущие `<Folder>` остаются валидными. Старый KMZ v1 читается v2-просмотрщиком без изменений, новый KMZ v2 читается v1-просмотрщиком через ваши же fallback’и (§3.6 — всё неизвестное идёт в ОКС/Пояснения).
- Добавляются **шесть новых префиксов** `cad_room_*` / `cad_str_*` / `cad_ons_*` / `cad_bu_*` / `cad_eq_*` / `cad_ben_*` и семь новых `<Folder>` (порядок строгий).
- Фото без EXIF-GPS получают детерминированную геометрию по **спирали Фибоначчи** вокруг центроида привязанного КН (раньше такие фото мы просто отбрасывали).
- Появляется параллельный канал `<ExtendedData>` — для текущей версии просмотрщика это no-op (вы сами пишете в §1, что ExtendedData не читаете), но он даёт машиночитаемый якорь для будущего рефакторинга 47a.

Приёмка может быть поэтапной: **P0** — ничего не делать, файл просто откроется, новые объекты упадут в ОКС/Пояснения через ваш существующий fallback; **P1** — добавить распознавание шести новых префиксов и парсинг ключей `Ссылка_документ_<i>` / `Ссылка_фото_<i>` для миниатюр в карточке; **P2** — открыть `graph.html` из архива как отдельную вкладку UI. Детали — в §A.8.

Терминологию, нумерацию ABGR-цветов, стиль таблиц и формулировки ключей мы постарались выдержать максимально близко к вашему документу.

## A.0 TL;DR — что меняется

1. **Шесть новых префиксов `styleUrl` / `Style id`:** `cad_room_*`, `cad_str_*`, `cad_ons_*`, `cad_bu_*`, `cad_eq_*`, `cad_ben_*`. Префиксы `cad_zu_*` / `cad_oks_*` / `cad_exp_*` / `photoPin_*` сохраняются без изменений.
2. **10 `<Folder>` верхнего уровня** вместо текущих 3 (порядок строгий, см. §A.3). Внутри «Фотографии» — четыре подпапки по физическому источнику.
3. **`<description>` — те же пары `Ключ: значение; `** (разделитель `; ` — точка-с-запятой + пробел, без HTML, без `<img>`) + фиксированный набор ключей для каждого нового класса (§A.5).
4. **`photoPin_*`** (§8 текущего контракта) теперь обязателен для **всех** фотографий. Никаких `<img>` в `<description>` мы больше не пишем.
5. **Фото без EXIF-GPS** располагаются на **спирали Фибоначчи** вокруг центроида привязанного КН с пустым центром r0 = 25 м, золотым углом φ = 137.508° и радиальным шагом r(i) = r0 + 4·√(i+1) м (§A.7). Координаты округляются до 7 знаков — побитово идентичный KMZ при повторной сборке.
6. **3D-extrude** для `cad_oks_*` / `cad_room_*` / `cad_ons_*`: `<extrude>1</extrude>` + `<altitudeMode>relativeToGround</altitudeMode>` + Z = `этажность × 3 м` в координатах. Для 2D-просмотрщика (Leaflet игнорирует 3-й элемент `<coordinates>`) изменений не требуется. Для Google Earth Pro даёт корректные объёмы (§A.6).
7. **`ExtendedData`** дублирует ключевые поля (`object_type`, `cad_number`, `z_meters_top`, `parent_cad`, `bu_id`, `ben_inn`, `floors_above`, `schema_version`, `z_source`). Для текущего контракта no-op, для «рефакторинга 47a» — машиночитаемый канал (§A.9).
8. **`<atom:author>` + `kml_schema_version=2.0`** в `<Document>`. Атрибут `xmlns:atom="http://www.w3.org/2005/Atom"` добавляется в корневой `<kml>`. Версия канала позволяет вам в будущем включать P1/P2-поведение только для v2.0+ (§A.10).
9. **Запросы к фронту:** (P1) парсить ключи `Ссылка_документ_<i>` / `Ссылка_фото_<i>` для миниатюр в карточке через JSZip + `URL.createObjectURL(blob)`; (P1) открыть `graph.html` из архива как третью вкладку UI «Граф связей» через `<iframe srcdoc>` или `<iframe src=blob:>` (§A.8).

Если по какой-то из девяти позиций есть возражение или встречное предложение — мы откатимся к compatibility-режиму (v1-стиль, всё в `cad_oks_*` / `cad_exp_*`) и пересоберём KMZ за один прогон парсера.

## A.1 Контекст — откуда взялись новые классы

Ниже — соответствие классов объектов из нашей модели `structure.json` (выход `03_nspd_load.py` и `04_nspd_graph.py`) и предлагаемых префиксов KML.

| Класс | Источник в `structure.json` | Геометрия | Префикс |
|---|---|---|---|
| ЗУ | `cadastre_objects[]` где `object_type` ~ «земельн» | Polygon (NSPD) | `cad_zu_*` (существующий) |
| ОКС-Здание | `cadastre_objects[]` где `object_type` ~ «здан» | Polygon (контур) | `cad_oks_*` (только здания) |
| Сооружение | `object_type` ~ «сооруж» | Polygon / LineString | `cad_str_*` (новый) |
| Помещение | `object_type` ~ «помещ|квартир|нежилое помещ|машино-мест» | Point (центроид здания) + Z | `cad_room_*` (новый) |
| ОНС | `object_type` ~ «незаверш» | Polygon | `cad_ons_*` (новый) |
| Бизнес-единица | `business_units[]` | Point (центроид первого КН) | `cad_bu_*` (новый) |
| Оборудование | `equipment[]` с `links.cadastre_id` + `links.level_ids` | Point на центроиде КН + z = уровень × 3 м | `cad_eq_*` (новый) |
| Бенефициар | `business_units[].owners[]` + ЕГРЮЛ/ЕГРИП | Point на юр.адресе (геокод) или Placemark без `<Point>` | `cad_ben_*` (новый) |

«Помещение» в текущем релизе — Point внутри родительского здания (центроид контура), с Z по этажу. Собственный контур помещения мы не пишем (см. §A.12) — если в NSPD появятся такие данные, мы расширим контракт точечно.

«Оборудование» — Point на центроиде привязанного КН с Z = `(level_idx − 1) × 3 м`, где `level_idx` — индекс этажа в нашей шкале (1-based, подвалы — отрицательные). Если оборудование привязано к диапазону этажей, берётся центральный.

«Бенефициар» без удачного геокода юр.адреса попадает в KML как Placemark **без** `<Point>` — наш парсер пишет такие записи, чтобы дедуп по ИНН работал даже без координат. Если ваш просмотрщик такие Placemark игнорирует — это допустимо (мы их в «Бенефициары» Folder складываем как plain-list).

## A.2 Новые префиксы `styleUrl` / `Style id`

Префикс — единственный сигнал классификатору, как и в v2.9.62 §3. ABGR ≠ RGBA (порядок байт: alpha, blue, green, red).

| Префикс | Класс | Геометрия | Формат `id` | LineStyle ABGR | PolyStyle ABGR | IconStyle ABGR |
|---|---|---|---|---|---|---|
| `cad_zu_*` | ЗУ *(существующий)* | Polygon | `cad_zu_<КН с _>` | `ff007f00` | `33007f00` | — |
| `cad_oks_*` | **Только здания** | Polygon | `cad_oks_<КН с _>` | `ff7f007f` | `337f007f` | — |
| `cad_room_*` | Помещение | Point | `cad_room_<КН с _>` | `ffaa00aa` | `44aa00aa` | `ffaa00aa` |
| `cad_str_*` | Сооружение | Polygon / LineString | `cad_str_<КН с _>` | `ff999999` | `66999999` | — |
| `cad_ons_*` | ОНС | Polygon | `cad_ons_<КН с _>` | `ff00ffff` | `4400ffff` | — |
| `cad_bu_*` | БУ | Point | `cad_bu_<slug>` | — | — | `ff1ea0ff` |
| `cad_eq_*` | Оборудование | Point | `cad_eq_inv_<инв.№>` или `cad_eq_uid_<sha1[:8]>` | — | — | `ff00aaff` |
| `cad_ben_*` | Бенефициар | Point | `cad_ben_inn_<ИНН>` или `cad_ben_innfl_<ИНН ФЛ>` | — | — | `ffe5541e` |
| `photoPin_*` | Фото *(существующий, §8 v2.9.62)* | Point | `photoPin_<КН с _>_<i>` или `photoPin_loose_<hash8>_<i>` | — | — | `ffff8800` |
| `cad_exp_*` | Пояснение *(существующий)* | Point | `cad_exp_<slug>` | — | — | `ffe5541e` |

В `<КН с _>` двоеточия заменяются на подчёркивания (как в вашем эталоне §2: `cad_zu_47_14_1203001_1`). `<slug>` для БУ — транслит названия в нижнем регистре с заменой не-ASCII на `_` и обрезкой до 32 символов. `<sha1[:8]>` — первые 8 hex-символов SHA1 от стабильного ключа (для оборудования без инв.№ — от пары `name + parent_cad`).

**Расширенная логика классификации (приоритет сверху вниз).** Этот список — предлагаемое продолжение §3 v2.9.62. Первые три пункта (1, 2, 3) уже у вас есть; пункты 4–9 — новые; пункты 10–13 — ваши существующие fallback’и.

1. `styleUrl` содержит `cad_zu_` → ЗУ.
2. `cad_exp_` → Пояснение.
3. `cad_oks_` → ОКС (в v2.10 — только здания).
4. `cad_room_` → Помещение.
5. `cad_str_` → Сооружение.
6. `cad_ons_` → ОНС.
7. `cad_bu_` → Бизнес-единица.
8. `cad_eq_` → Оборудование.
9. `cad_ben_` → Бенефициар.
10. `photoPin` / `photo_icon` / `photo_` → Фото.
11. *(fallback)* «земл» в name/description (граница слова) → ЗУ.
12. *(fallback)* геометрия Point → Пояснение.
13. *(иначе)* → ОКС.

Если P1 у вас не реализован, пункты 4–9 не отрабатывают, и новые объекты падают в пункт 13 (ОКС) или 12 (Пояснения) — это допустимое поведение, дедуп всё равно сработает по кад.№ из `<name>`.

## A.4 `<name>` Placemark — шаблоны

Кадастровый номер должен быть **отдельным токеном** в `<name>` (правило §0.2 v2.9.62): `\b\d{2}:\d{2}:\d{1,8}:\d{1,8}\b`. Дублируется в `<description>` как `Кадастровый номер: <КН>;`.

| Класс | Шаблон `<name>` | Пример |

Значение Z вычисляется как `floors_above × 3 м`. Источник — `MENTAL_CHECK_REPORT.md §1` (высота этажа стандартизирована на 3 метра).

| Уровень | Z, м |
|---|---|
| Этаж 1 | 0 |
| Этаж 2 | 3 |
| Этаж 3 | 6 |
| Этаж 4 | 9 |
| Подвал 1 | −3 |
| Подвал 2 | −6 |
| Мансарда (4-этажный дом) | 12 |

**В Leaflet 3-й элемент `<coordinates>` игнорируется** — никаких изменений у фронта не требуется, объекты по-прежнему рисуются плоскими. Для Google Earth Pro даёт корректные 3D-объёмы (полезно для презентаций владельцу).

**`innerBoundaryIs` (дырки в полигонах) НЕ используется** — несовместимо с §4 v2.9.62, который требует одну `LinearRing` на `<Polygon>`. Если у Polygon в исходных данных есть дырки (что встречается в NSPD редко), они пропускаются с warning в логе парсера.

## A.7 Спираль Фибоначчи для фото без EXIF-GPS

### Геометрия

Пусть `(lon0, lat0)` — центроид привязанного КН. Для i-го фото (i = 0, 1, 2, …, отсортированы по `basename` в стабильном порядке):

```
φ_i (рад) = i × (137.508° × π/180)
r_i (м)   = 25.0 + 4.0 · √(i+1)
dx (м)    = r_i · cos(φ_i)
dy (м)    = r_i · sin(φ_i)
Δlon (°)  = dx / (111320 · cos(lat0_rad))
Δlat (°)  = dy / 111320
lon_i     = lon0 + Δlon
lat_i     = lat0 + Δlat
alt_i     = 0
```

Константы:

- `r0 = 25.0` м — пустой центр (не накрываем сам объект).
- `dr = 4.0` м — радиальный шаг спирали.
- `φ_step = 137.508°` — золотой угол (Vogel’s spiral). Гарантирует, что точки распределены равномерно, без видимых лучей.
- `m_per_deg_lat = 111320.0` м/° — эталонная константа WGS84 (для широт 40–60° погрешность < 0.5%).

### Центроид

Источник `(lon0, lat0)` зависит от типа геометрии родительского объекта:

- **Polygon** → среднее (по координатам) всех вершин внешнего контура без замыкающей точки.
- **Point** → сам Point.
- **LineString** → среднее всех вершин.
- **MultiPolygon** → центроид первого полигона.

### Эталонная реализация (Python 3.13)

```python
import math

GOLDEN_DEG    = 137.508
R0_M          = 25.0
DR_M          = 4.0
LAT_M_PER_DEG = 111320.0


def spiral_points(lat0: float, lon0: float, count: int,
                  r0: float = R0_M, dr_factor: float = DR_M,
                  golden_angle: float = GOLDEN_DEG
                  ) -> list[tuple[float, float, float]]:
    """Спираль Фибоначчи вокруг (lat0, lon0) с пустым центром r0.

    Возвращает count кортежей (lat, lon, alt=0.0). Детерминирована:
    при одинаковых аргументах — побитово идентичный вывод.

    Алгоритм:
      r_i   = r0 + dr_factor · √(i+1)
      phi_i = i · golden_angle (в радианах)
      dx, dy = r_i · cos(phi_i), r_i · sin(phi_i)
      d_lon = dx / (111320 · cos(lat0_rad))
      d_lat = dy / 111320
    """
    if count <= 0:
        return []
    lat0_rad      = math.radians(lat0)
    m_per_deg_lon = max(LAT_M_PER_DEG * math.cos(lat0_rad), 1.0)
    phi_step_rad  = math.radians(golden_angle)
    out: list[tuple[float, float, float]] = []
    for i in range(count):
        phi = i * phi_step_rad
        r   = r0 + dr_factor * math.sqrt(i + 1)
        dx, dy = r * math.cos(phi), r * math.sin(phi)
        d_lon  = dx / m_per_deg_lon
        d_lat  = dy / LAT_M_PER_DEG
        out.append((round(lat0 + d_lat, 7),
                    round(lon0 + d_lon, 7), 0.0))
    return out
```

### Таблица первых 8 точек вокруг (lat0=47.0, lon0=39.0)

Для проверки реализации фронтом (если захотите визуализировать spiral-облако):

| i | r_i, м | φ_i, ° (mod 360) | dx, м | dy, м | lat_i | lon_i |
|---|---|---|---|---|---|---|
| 0 | 29.000 | 0.000 | 29.000 | 0.000 | 47.0000000 | 39.0003806 |
| 1 | 30.657 | 137.508 | −22.624 | 20.708 | 47.0001860 | 38.9997031 |
| 2 | 31.928 | 275.016 | 2.769 | −31.808 | 46.9997143 | 39.0000363 |
| 3 | 33.000 | 52.524 | 20.066 | 26.196 | 47.0002353 | 39.0002634 |
| 4 | 33.944 | 190.032 | −33.404 | −5.917 | 46.9999469 | 38.9995617 |
| 5 | 34.795 | 327.540 | 29.336 | −18.640 | 46.9998325 | 39.0003850 |
| 6 | 35.578 | 105.048 | −9.218 | 34.366 | 47.0003087 | 38.9998790 |

Радиальная гистограмма для первых 100 точек: минимум ≈ 29 м (i=0), максимум ≈ 65 м (i=99), плотность убывает как 1/r — характерная картина спирали Vogel.

### Детерминированность

- Фото внутри одного КН сортируются по `basename` (lexicographic, case-sensitive). Это гарантирует, что i=0 всегда одно и то же фото при повторных сборках.
- Координаты округляются до 7 знаков после запятой (~1 см точности — избыточно, но позволяет получить побитово идентичный KMZ).
- Константы r0/dr/φ_step зашиты в код и не зависят от системы пользователя.

Итого — два прогона `08_build_kmz_v2.py` на одинаковом `_data/structure.json` дают одинаковый `sha256(project.kmz)`.

### Метаданные в `<description>`

Для каждого spiral-фото в description добавляются три ключа:

```

```

Это позволит вашему просмотрщику (если захотите) отрисовать spiral-точки полупрозрачным значком (`opacity: 0.6`) или показать в карточке отметку «синтетические координаты».

- Каждый файл `images/<f>` упомянут **ровно одним** `photoPin_*` Placemark (значение в `<coordinates>`).
- Каждый файл `docs/<f>` упомянут **одним или несколькими** Placemark через ключ `Ссылка_документ_<i>: docs/<f>` в `<description>` (i от 1).
- `graph.html` — статический inline HTML, не зависит от внешних ресурсов (CDN, картинок, JS). Готов открываться через `iframe srcdoc` или `iframe src=blob:`.
- ZIP-метаданные нормализованы: `zinfo_date=(2025,1,1,0,0,0)`, `compress_type=ZIP_DEFLATED` — для идемпотентности sha256.

**Запросы к фронту (поэтапно):**

| Этап | Что требуется | Когда |
|---|---|---|
| **P0** | Игнорировать `docs/`, `graph.html` без сбоев. Чтение KMZ как ZIP, разбор `doc.kml`. | Работает уже в v2.9.62 — действий не требуется. |
| **P1** | Парсить ключи `Ссылка_документ_<i>` / `Ссылка_фото_<i>` из `<description>` и отрисовывать миниатюры в карточке через `JSZip` + `URL.createObjectURL(blob)`. | К приёмке v2.10.0. |
| **P1** | Открыть `graph.html` из архива как **третью вкладку UI** («Граф связей») через `<iframe srcdoc>` (предпочтительно — изолированный контекст) или `<iframe src=blob:>` (если srcdoc не подходит из-за размера). | К приёмке v2.10.0. |
| **P2** | Добавить Folder→тип как вторичный якорь к §3 классификатора. Если префикс `cad_*` потерян (повреждён файл), но Placemark лежит в Folder с именем «Помещения» — отнести к Помещениям. | Опционально, к v2.11. |

Если P1 не успевает в спринт — это не блокер. KMZ останется валидным, объекты на карте будут видны, просто без миниатюр и без вкладки графа. P0-поведение гарантировано.

## A.9 `ExtendedData` — параллельный канал

В каждом Placemark добавляется блок `<ExtendedData>` с машиночитаемыми полями. Для текущего просмотрщика v2.9.62 это no-op (§1 вашего контракта: «ExtendedData / Data / SimpleData сейчас НЕ читаются»), но для будущего рефакторинга 47a — единственный надёжный источник типа объекта.

```xml
<ExtendedData>
  <Data name="object_type"><value>room</value></Data>
  <Data name="cad_number"><value>61:44:0050706:119</value></Data>
  <Data name="parent_cad"><value>61:44:0050706:31</value></Data>
  <Data name="bu_id"><value>filial_rostov</value></Data>
  <Data name="ben_inn"><value>6164012345</value></Data>
  <Data name="z_meters_top"><value>9.0</value></Data>
  <Data name="z_meters_bottom"><value>0.0</value></Data>
  <Data name="floors_above"><value>4</value></Data>
  <Data name="schema_version"><value>2.0</value></Data>
  <Data name="z_source"><value>level</value></Data>
</ExtendedData>
```

Возможные значения `object_type`: `zu`, `oks`, `room`, `structure`, `ons`, `business_unit`, `equipment`, `beneficiary`, `photo`, `explanation`.

Возможные значения `z_source`: `level` (этаж из NSPD/ЕГРН), `floors_above` (вывод из этажности здания), `spiral` (для фото), `unknown`.

Поле `schema_version=2.0` позволит вашему просмотрщику в будущем переключаться на «строгий режим» (читать тип только из `<ExtendedData>`, без эвристик по префиксу) — но это уже задача 47a, не v2.10.

## A.10 Версионирование

В корневом `<kml>` добавляется namespace `atom`:

```xml
<kml xmlns="http://www.opengis.net/kml/2.2"
     xmlns:atom="http://www.w3.org/2005/Atom">
```

В `<Document>` — блок автора и версии генератора:

```xml
<atom:author>
  <atom:name>pirushin_sosn_rocha_08_build_kmz v2</atom:name>
</atom:author>
<ExtendedData>
  <Data name="kml_schema_version"><value>2.0</value></Data>
  <Data name="generator"><value>pirushin_sosn_rocha_08_build_kmz_v2.py</value></Data>
  <Data name="generated_at"><value>2026-05-18T14:23:11Z</value></Data>
</ExtendedData>
```

`generated_at` — UTC ISO-8601 с точностью до секунды; если генератор запущен в детерминированном режиме (`--deterministic`), значение фиксируется на эпохе `2025-01-01T00:00:00Z` для воспроизводимости.

**Совместимость:**

- **v2 KMZ → v1 (v2.9.62) просмотрщик:** новые префиксы `cad_room_*`, `cad_str_*`, `cad_ons_*` упадут в fallback §3.6 (ОКС). `cad_bu_*`, `cad_eq_*`, `cad_ben_*` — в fallback §3.12 (Point → Пояснение). `<ExtendedData>` игнорируется. Карта по-прежнему рисуется, дедуп по кад.№ работает.
- **v1 KMZ → v2 просмотрщик (после P1):** всё работает по существующим fallback’ам. Никаких изменений.
- **v2 KMZ → v2 просмотрщик (после P1):** полное распознавание всех 10 классов, миниатюры, вкладка графа.

Версия канала `kml_schema_version` позволит вам в будущем включать строгое поведение (P2) только для файлов 2.0+ — без риска сломать обработку старых KMZ.

## A.11 Чек-лист валидации (расширенный)

Это расширение вашего чек-листа из §10 v2.9.62. Пункты, отмеченные звёздочкой, — новые.

- [ ] `<styleUrl>` у каждого Placemark начинается с одного из 10 префиксов: `#cad_zu_`, `#cad_oks_`, `#cad_room_`*, `#cad_str_`*, `#cad_ons_`*, `#cad_bu_`*, `#cad_eq_`*, `#cad_ben_`*, `#photoPin_`, `#cad_exp_`.
- [ ] `<Style id>` определён, id уникален в документе, ≤ 64 символов, ASCII-safe.
- [ ] Кад.№ (где есть) — отдельный токен `\b\d{2}:\d{2}:\d{1,8}:\d{1,8}\b` в `<name>`, продублирован в `<description>` ключом `Кадастровый номер:`.
- [ ] `<description>` — пары `Ключ: значение`, разделитель `; ` (точка-с-запятой + пробел). **Без HTML, без `<img>`, без переносов.**
- [ ] Координаты в формате `lon,lat,alt`; полигон замкнут (первая = последняя), ≥ 4 точки; WGS84, ≥ 6 знаков после запятой.
- [ ] Одна геометрия на Placemark; нет `<innerBoundaryIs>`; нет `<MultiGeometry>`.
- [ ] 10 `<Folder>` в фиксированном порядке (§A.3); префикс styleUrl ↔ Folder согласованы.
- [ ] *Для `cad_oks_*` / `cad_room_*` / `cad_ons_*`: `<extrude>1</extrude>` + `<altitudeMode>relativeToGround</altitudeMode>` + ненулевой Z в `<coordinates>`.
- [ ] *Для `photoPin_*` без EXIF-GPS: координаты по спирали (r0=25 м, φ=137.508°, r=r0+4√(i+1) м); в `<description>` есть ключ `Источник координат: spiral_around_centroid` + `r_meters` + `phi_deg`.
- [ ] UTF-8 без BOM; кириллица в CDATA; XML валиден по `xmllint --noout`.
- [ ] Все `images/<f>` и `docs/<f>`, упомянутые в description, физически присутствуют в архиве; `graph.html` в корне архива (если был на входе генератору).
- [ ] *`<ExtendedData kml_schema_version=2.0>` присутствует в `<Document>`; в каждом Placemark — как минимум `object_type` и (где применимо) `cad_number` / `z_meters_top` / `bu_id` / `ben_inn`.
- [ ] *`<atom:author>` присутствует в `<Document>`; `xmlns:atom` объявлен в корневом `<kml>`.
- [ ] *Идемпотентность: два прогона генератора на одинаковом входе → одинаковый `sha256(project.kmz)`.

Если вы хотите получить эталонный smoke-KMZ для интеграционного теста — мы можем выложить его в `tests/fixtures/golden_v2.10.kmz` (синтетический проект: 1 ЗУ + 1 здание + 2 помещения + 1 сооружение + 1 ОНС + 1 БУ + 2 оборудования + 1 бенефициар + 5 фото с GPS + 3 фото без GPS). Скажите — пришлём.

## A.12 Открытые вопросы / на будущее

Эти пункты мы намеренно вынесли за рамки v2.10 — чтобы не раздувать спецификацию и согласовать отдельно. Если у вас есть мнение или предпочтение по любому из них — напишите, мы заложим в v2.11 или раньше.

1. **`<TimeStamp>` / `<TimeSpan>`.** Сейчас пишем `Снято: YYYY-MM-DD HH:MM:SS` только в `<description>` фото. Если ваш просмотрщик планирует слайдер времени или фильтр «фото за последний месяц», мы можем добавить полноценный `<TimeStamp><when>` в каждый `photoPin_*` Placemark. По запросу.
2. **`<NetworkLink>`.** Динамическая подгрузка тайлов WMS nspd.gov.ru через KML-`NetworkLink` мы не используем — у вас в sw-cache уже всё корректно сделано на стороне приложения. Если когда-нибудь понадобится прицепить внешний слой (например, кадастровая подложка из стороннего источника) — обсудим отдельно.
3. **Кастомные иконки от фронта.** Сейчас IconStyle ссылается на стандартный набор `http://maps.google.com/mapfiles/kml/shapes/*.png` (`donut`, `target`, `forbidden`, `man`, `camera`). Если у вас есть собственный icon-pack (например, в `assets/icons/`), пришлите список — мы переключим IconStyle на относительные пути внутри KMZ (`<href>icons/room.png</href>`).
4. **Помещения с собственным контуром.** Сейчас `cad_room_*` — Point внутри родительского здания. Если в NSPD появятся данные о полигональных контурах помещений (что иногда встречается для нежилых помещений в БЦ), мы готовы расширить контракт до Polygon + Z. Геометрия и `<description>` останутся теми же, добавится только опциональный Polygon.
5. **Маршруты / тайм-серии (треки).** Если в проекте появятся данные о маршрутах обходов, трассах кабелей или временных рядах перемещений оборудования, нужен будет новый префикс `cad_track_*` с `LineString` + `<TimeSpan>`. Обсудим, когда задача появится.
6. **Синхронизация версий контракта.** Предлагаем держать пару `parser_version` ↔ `viewer_version` в одном месте — например, в файле `docs/CONTRACT_VERSIONS.md` в репозитории просмотрщика. На сегодня:
   - `parser v2.10.0` ↔ `viewer v2.9.62` — режим compatibility (P0).
   - `parser v2.10.0` ↔ `viewer v2.10.0+` — полная функциональность (P1 + P2).
7. **Способ обратной связи.** Предпочитаемый канал для уточнений по этой спецификации — issue в репозитории просмотрщика `romanbabenkorostov-ux/ekcelo` с меткой `kml-spec`. Если issue неудобен — пишите на e-mail команды парсера (адрес отдельным письмом). Срочные вопросы (блокеры приёмки) — мессенджер.
8. **Гранулярность дедупа.** В v2.9.62 §7 дедуп — по кад.№, затем по `name||description`, затем по округлённой геометрии. Для `cad_eq_*` и `cad_ben_*` кад.№ отсутствует — рекомендуем использовать инв.№ и ИНН соответственно (они уже в `<ExtendedData>` как `cad_number` ↔ заменяющие ключи `bu_id` / `ben_inn`). Если нужен дополнительный helper-ключ в `<description>` — скажите.
