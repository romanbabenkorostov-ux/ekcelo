# ADR-001: Расширение схемы БД для профиля экспорта в ЭТП

**Статус:** Accepted
**Дата:** 2026-05-27
**Связано:** SPEC `docs/etp_export/SPEC_etp_export.md` §5, PR #41 (merged, SHA `8e1272f`), PR #48 (merged — ADR-001 Proposed), PR #50/52 (CORRESPONDENCE/025+026 ratification), миграция `schema/migrations/0001_etp_profile.sql`.

---

## Контекст

`CLAUDE.md` фиксирует принцип: «БД = слепок ЕГРН» (`schema/egrn_current_schema.sql`). Все таблицы содержат поля, происходящие из выписок ЕГРН и обогащения NSPD; ручные правки в БД не предполагаются.

Для экспорта объектов сюрвея в ЭТП (`torgi.gov.ru`, `roseltorg.ru`, `sberbank-ast.ru`) спеке требуется набор полей, которых в ЕГРН **нет** (см. §10 SPEC, 12 пунктов): окружение и транспорт, степень износа, инженерные сети, отделка, риски, преимущества, тип процедуры (банкротство/приватизация) и т.д. Эти поля заполняются экономистом (ОСВ/survey-лист) или парсером EXIF UserComment из фото объекта.

Если хранить эти данные в существующих таблицах ЕГРН, теряется принцип «БД = слепок» — нельзя будет пересоздать БД из выписок без потерь. Альтернативные варианты:

1. **Manual-overlay JSON-файлы** рядом с БД (sidecar). Сохраняет принцип, но мерж в экспортёре превращается в отдельный пайплайн с риском рассинхрона по КН.
2. **Только NSPD/LLM** автоматически. Сохраняет принцип, но качество описаний низкое; экономист не может ручно править.
3. **Отдельные таблицы** в той же БД, явно помеченные как «не-ЕГРН». Нарушает принцип буквально, но позволяет реляционно связать с `objects`, держит транзакционность и индексы.

## Решение

Принимаем **вариант 3** — отдельные таблицы в `schema/egrn_current_schema.sql`, явно отделённые от ядра ЕГРН по неймингу и комментарию:

- `object_etp_profile (cad_number, location_extra, building_extra, layout, legal_extra, risks, extras, source, confidence, updated_at)` — JSON-колонки, FK на `objects(cad_number)`.
- `lots (lot_id, name, platform_targets, procedure_type, deal_type, primary_cad_number, notes_md, created_at)`.
- `lot_items (lot_id, cad_number, role, ord)` — many-to-many.

Принцип переформулируется (правка `CLAUDE.md` идёт вместе с миграцией):

> **БД = слепок ЕГРН + ЭТП-профиль.** Таблицы с префиксом `object_etp_*`, `lots*` явно содержат не-ЕГРН данные (ручные правки экономиста, EXIF-выводы, LLM-обогащение). Каждое поле имеет `source` ∈ {`osv`, `exif`, `manual`, `nspd`, `llm`} и `confidence`. При пересоздании БД из выписок ЭТП-профиль НЕ восстанавливается — это отдельный слой.

## Последствия

### Положительные
- Реляционная целостность (FK к `objects`), транзакции, индексы из коробки.
- Экспортёр читает БД одним запросом, без мерж-пайплайна с sidecar-файлами.
- Версионирование через миграции (`schema/migrations/0NN_*.sql`) единообразно с остальной схемой.
- Поле `source/confidence` делает происхождение каждого значения явным — viewer может отображать бейджи доверия.

### Отрицательные / риски
- Принцип «слепок ЕГРН» больше не буквальный — нужна явная документация и в `CLAUDE.md`, и в README схемы.
- При восстановлении БД из выписок ЭТП-профиль теряется. Митигация: ETL из ОСВ/XLSX и парсер EXIF могут восстановить большую часть; ручные нотатки экономиста хранятся в `lots.notes_md` и должны бэкапиться отдельно (документ-overlay в `documents/`).
- Дополнительные ~30 полей в БД увеличивают сложность viewer (нужны UI для отображения и редактирования).

### Нейтральные
- JSON-колонки vs нормальные колонки: выбрали JSON (см. SPEC §5, ответ на вопрос «миграция схемы БД»), даёт гибкость для расширения без новых миграций; теряем строгую типизацию и индексы по вложенным полям. На текущем масштабе (< 10 000 объектов) приемлемо.

## Альтернативы (rejected)

| Вариант | Почему отклонён |
|---|---|
| Manual-overlay JSON sidecar | Дублирует ключ КН в двух источниках; синхронизация — отдельный риск; нет транзакций. |
| Только NSPD + LLM | Нет ручного контроля экономиста; качество описаний для ЭТП низкое; LLM-галлюцинации в юридически значимом тексте. |
| Расширение `objects` новыми колонками | Размывает семантику таблицы; принцип «слепок ЕГРН» теряется и явно, и неявно. |
