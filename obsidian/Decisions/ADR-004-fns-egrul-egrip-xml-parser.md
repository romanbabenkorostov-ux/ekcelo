# ADR-004: Парсер выписок ФНС ЕГРЮЛ/ЕГРИП (XML) как официальный источник данных о субъектах

**Статус:** Accepted · **Дата:** 2026-06-05 · **Автор:** parser-team
**Связанные:** [[parallel-parsers-map]], [[ADR-001-etp-profile-extension]], [[ADR-002-parser-checko-integration-policy]]

## Контекст

Данные о юрлицах/ИП (руководители, учредители, доли, реорганизация, статус,
ОКВЭД) приходят в ekcelo из нескольких источников:

- **ФНС XML-выписки** ЕГРЮЛ/ЕГРИП — официальные машиночитаемые файлы (заказчик
  приложил XSD: ЕГРЮЛ v4.08 `VO_RUGF_2_311_26_04_08_01`, ЕГРИП v4.07
  `VO_RIGF_2_311_27_04_07_01`). Без API-ключей и secrets, кодировка windows-1251,
  без targetNamespace.
- **checko.ru / dadata.ru** — API (ADR-002, интеграция отложена, нужны secrets).
- **PDF-выписки** — то, что чаще всего есть на руках у экономиста
  (`E:\Code\ekcelo\primer_for_parsing\EGRUL-EGRIP`).

Нужно: единая точка приёма этих данных, версионирование форматов ФНС и
автоопределение, без дублирования логики на каждый источник.

## Решение

1. **Добавлен парсер `egrn_parser/parsers/egrul_egrip_parser.py`** — ФНС-XML
   (ЕГРЮЛ/ЕГРИП). Это отдельный домен от Росреестр-ЕГРН (`xml_parser.py`).
2. **Автоопределение формата** по корню `Файл`: реестр — из `@ТипИнф`
   (`ЕГРЮЛ_*` / `ЕГРИП_*`), версия — из `@ВерсФорм`. Поддержанные пары —
   `SUPPORTED_FORMATS`; неизвестная версия → `ValueError` (а не молчаливый разбор).
3. **XSD собираются по реестрам** в `parser/schema/xsd/{egrul,egrip}/`
   (по конвенции `xsd/upd`): оригиналы cp1251 + `NOTES.md`, новые редакции
   кладутся рядом, берётся свежайшая по сортировке имени. Валидация — через lxml
   (опциональна, не падает без неё).
4. **Единая нормализованная запись** (одна на `Документ`), общая для ВСЕХ
   источников: `{subject, directors, managing_orgs, founders, predecessors,
   successors, source}`. checko/dadata-JSON и PDF мапятся в ту же форму →
   downstream (запись в БД, граф) не знает про источник.
5. **ФНС-XML — приоритетный источник** там, где выписка есть (официальный,
   без secrets). checko/dadata/PDF — gap-fill (как приоритеты `source` в §6,
   ADR-001/ADR-002).

## Куда ложатся данные (связь с ADR-001)

ЕГРЮЛ/ЕГРИП — это НЕ Росреестр-ЕГРН (§1–§5 = слепок объектов недвижимости).
Данные о субъектах относятся к не-ЕГРН слою: `entity_registry`
(ИНН/ОГРН/наименование/тип) + `object_etp_profile.legal_extra`
(статус/ОКВЭД/форма). Запись помечается `source='fns_egrul'`/`'fns_egrip'`
с высоким `confidence` (официальный источник). При пересоздании БД из выписок
Росреестра этот слой НЕ восстанавливается (ADR-001).

**Враппер «нормализованная запись → БД» НЕ реализован** в этой итерации:
целевая схема subjects/relations проектируется в соседнем чате
(`contracts/db/SCHEMA_SPEC.md`). Парсер отдаёт dict; запись добавится, когда
схема будет готова. Никаких спекулятивных таблиц — Simplicity First (CLAUDE.md).

## Альтернативы (rejected)

- **Парсить ФНС-XML внутри `xml_parser.py`** — смешение доменов (Росреестр vs ФНС),
  разные корни/неймспейсы. Отклонено.
- **Свой парсер на каждый источник без общей записи** — дублирование downstream-логики
  и расхождение полей. Отклонено в пользу единой нормализованной записи.
- **Хардкод одной версии формата** — ломается на новой редакции ФНС молча.
  Отклонено в пользу `SUPPORTED_FORMATS` + явный `ValueError`.

## Последствия

- ✅ Официальный источник данных о субъектах без secrets (в отличие от checko).
- ✅ Версионирование ФНС-форматов управляемо (добавить пару + XSD рядом).
- ✅ Единая запись → checko/dadata/PDF подключаются адаптерами без новых таблиц.
- ⚠️ Запись в БД ждёт `contracts/db/SCHEMA_SPEC.md` (соседний чат) — пока парсер
  даёт только нормализованный dict.
- ⚠️ PDF-источник у заказчика основной, а XML-выписок на руках нет → следующий
  приоритет — PDF-адаптер в ту же запись.
