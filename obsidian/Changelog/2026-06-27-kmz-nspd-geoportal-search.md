# 2026-06-27 — KMZ/NSPD: переход на geoportal-search (контур наконец парсится)

## Симптом
`kmz --nspd` ловил **0 feature**, хотя карта NSPD **показывала** контур после
клика по лупе («вывелся объект с контуром и описанием, но не спарсилось»).

## Диагноз (по сохранённой странице заказчика)
Страница = Vite module-federation SPA; сам map-бандл и XHR-ответ с геометрией в
«Save as» **не попали**. Но улика нашлась в логике v8: `NetworkCapture`
**пропускает любой URL c `/search/`** (правило «search = extent квартала» — из
эпохи ПКК). Современный NSPD отдаёт геометрию объекта именно эндпоинтом
`https://nspd.gov.ru/api/geoportal/v2/search/geoportal` (его и дёргает лупа) →
реальный ответ молча отбрасывался.

## Фикс — `geo_nspd_browser.py` переписан
- **Активный** session-aware запрос через `page.request.get` к geoportal-search
  (наследует cookies/Referer открытой карты → проходит анти-бот там, где голый
  urllib даёт 403). Варианты: `?query=КН` и `?thematicSearchId=1&query=КН`.
- Парс ответа терпим к обёрткам (`{data:{features}}` / FeatureCollection /
  Feature); КН берётся в т.ч. из `properties.options.cad_num`.
- Геометрия в EPSG:3857 → репроекция в WGS84 (inline, формула из v8).
- Доп. **разрешающий** слушатель ответов (НЕ режет `/search/geoportal`) копит
  feature карты → обнаружение ОКС в границах ЗУ (centroid-in-polygon) + fallback.
- Контракт `fetch_parcels` неизменен: `{cad:{polygon,buildings,captured}}`.

## CLI

## Версия 5: ручной режим `--nspd-manual` (человек в цикле)
Авто-клик лупы ОКС не вытянул (geoportal-search отдаёт только контур ЗУ; данные
вкладки «ОКС в пределах» NSPD грузит лишь при реальном взаимодействии). Решение
по идее заказчика — **человек в цикле**:
- `--nspd-manual` → принудительно видимый браузер; по каждому КН скрипт открывает
  страницу и **ждёт Enter в консоли**, пока пользователь жмёт лупу и листает
  вкладки (ОКС в пределах ЗУ).
- Ожидание Enter — через `loop.run_in_executor(None, input)` → event-loop НЕ
  блокируется, пассивный слушатель продолжает копить features во время кликов.
- Слушатель в ручном режиме ловит **любой** JSON c nspd (не только /geoportal/),
  т.к. данные вкладок идут с других путей; `extract_features` отсеет не-feature.
- После Enter — парсим перехваченное, ОКС фильтруем centroid-in-polygon, идём к
  следующему КН.

## Версия 7: найден эндпоинт списка ОКС (objectsList) — финал
Диагностика прогона показала точный эндпоинт списка ОКС в пределах ЗУ:
```
/api/geoportal/v1/tab-group-data?tabClass=objectsList&categoryId={cat}&geomId={gid}
```
(для ЗУ 23:15:0000000:2267 вернул **22 КН** ОКС). `geomId` и `categoryId` берутся
из самого feature ЗУ (`feature.id`, `properties.category`).

Рефактор (`_parcel_ids`, `_fetch_objects_list`, `_resolve_geom`, `_geoportal_get`):
1. По каждому ЗУ: contour через geoportal-search (3/3 работает).
2. **Прямой** запрос objectsList по geomId/categoryId → список КН ОКС (не нужно
   вручную листать вкладки; работает и в авто-режиме `--nspd`).
3. Геометрию каждого ОКС резолвим `_geoportal_get` (лёгкий, без ретраев). Есть
   контур → полигон; **нет контура → точка по спирали** (geometry=None) — ровно
   по ТЗ «если нет контуров — по спирали».
4. Прогресс: `[i/N] ЗУ … контур ✓`, `ОКС в пределах: N`, `j/N КН: контур|спираль`.
5. Удалён мёртвый `_trigger_lupa` (заменён прямым objectsList).
6. CLI печатает **абсолютный путь** KMZ: `✅ KMZ сохранён: <path>`.
- Тесты +3 (`_parcel_ids`): **geo browser 12 passed**.

Запуск (объекты грузятся автоматически, лупу жать не обязательно):
```
python -m egrn_parser kmz --parcels 23:15:0000000:2267,23:15:0303000:1562,23:15:0303000:1130 \
  --db проект.sqlite --out objects.kmz --nspd-headful
```
`--nspd-manual` остаётся как страховка (ручное листание вкладок добирается из перехвата).

## Версия 9: контуры ОКС где есть + кучная спираль
Прогон: 22 ОКС, но все ушли в спираль (контур 0). Доработки:
- `_resolve_geom` теперь принимает **Polygon → контур** И **Point → реальную
  точку** NSPD (точнее синтетической спирали); None → спираль. Поиск ОКС
  расширен тематикой `thematicSearchId=2` (ОКС), exact-cad match.
- Диагностика per-ЗУ: `→ контур: N, точка: M, спираль: K` — видно, что
  отдаёт NSPD по каждому ОКС.
- **Спираль переписана** с архимедовой на **филлотаксис (золотой угол)**: старая
  при целом числе оборотов вырождалась в линию (cos резонировал с числом точек —
  все x совпадали). Новая: `rad=√((i+½)/n)`, `θ=i·137.5°`, `margin=0.45` —
  кучно у центра ЗУ, без вырождения. Центроид считается без замыкающей вершины.
- Тесты: geo **29 passed**.

## Версия 10: поиск ОКС по всем темам NSPD (search-theme)
Прогон: все 22 ОКС → спираль (контур 0, точка 0). Причина: geoportal-search в
**дефолтной теме индексирует только ЗУ**; ОКС (квартал 0314001) там не находятся,
а thematicSearchId 1/2 — не их тема.
Фикс:
- `_fetch_search_themes` — GET `/api/geoportal/v1/search-theme?pageCode=geoportal`
  → список (id, название) всех тем поиска NSPD; печатается один раз в начале.
- `_geoportal_get(page, cad, theme_ids)` теперь перебирает `?query=` + по каждой
  теме `?thematicSearchId={id}&query=` → объект находится независимо от темы.
- Резолв ОКС и поиск ЗУ используют эти темы.
Диагностика покажет строку `[темы поиска NSPD] 1=…; …` — по ней видно тему ОКС;
если и при переборе всех тем контур 0 — значит NSPD не отдаёт геометрию ОКС
текстовым поиском, перейдём на геометрию по geomId из objectsList.

## Версия 11: geomId-проба геометрии ОКС (текст-поиск ОКС не находит)
Прогон с темами подтвердил: тема «1=Объекты недвижимости» включает ОКС (под ней
находится сам ЗУ), но **текстовый поиск 22 ОКС всё равно даёт 0** — значит эти
ОКС (ранее учтённые, квартал 0314001) не индексируются по КН. У них есть
**geomId** в ответе objectsList — геометрию надо брать по нему.
- `oks_records(data)` — из objectsList достаёт `{cad, geomId, categoryId}` на ОКС
  (раньше брались только КН-строки).
- `_probe_geom_by_id` — пробует кандидатные эндпоинты геометрии по geomId/cat
  (`geoportal/v1|v2/geom/{cat}/{gid}`, `…?categoryId&geomId`, WFS featureID) и
  печатает `status/features` по каждому → один прогон покажет рабочий эндпоинт.
- Диагностика печатает пробу для первого ОКС с geomId.
Спираль остаётся фолбэком, пока не зафиксируем рабочий geomId-эндпоинт.
- Тесты +2 (`oks_records`): geo browser **15 passed**.

## Версия 13: objectsList = плоский список КН (без geomId) → geomId из feature.id
Сырой objectsList оказался плоским списком КН без geomId:
`{"object":[{"title":"Объект недвижимости: ","value":["23:15:…", …]}]}`. А вторая
загруженная карточка дала тип сооружения: `selectedCard=415708564,36383,90:25:020103:9298`
→ **categoryId 36383 = сооружение** (36369 = здание).
Доработки:
- `extract_features(require_geometry=False)` — берём feature ОКС даже без
  геометрии (контур грузится лениво по geomId).
- `_oks_search` → feature ОКС (любая геометрия) + raw; `_feature_ids` → geomId
  (`feature.id`) + categoryId (`properties.category`).
- `_resolve_geom` переписан: search→feature; есть геометрия — берём; иначе
  geomId→`_resolve_geom_by_id` (перебор кандидатных эндпоинтов).
- Диагностика по первому ОКС: `[поиск ОКС …] feature/geometry/geomId/cat` +
  `[search raw]` + проба эндпоинтов по найденному geomId.
- SPEC обновлён (типы ОКС, плоский objectsList, geomId из feature.id).
- Тесты +2 (require_geometry, _feature_ids): geo browser **17 passed**.

## Версия 14: геометрия ОКС через НАВИГАЦИЮ карты (search ОКС пуст)
Диагностика подтвердила: `search/geoportal` по КН ОКС — **feature=нет** (вообще

  малый полигон внутри границы ЗУ (size-guard <30% площади ЗУ).
- Слушатель `_on_resp` расширен: ловит ЛЮБОЙ nspd-JSON (геометрия ОКС идёт не
  только с /geoportal/).

- SPEC обновлён. Цена: ~4 c/ОКС (22 ОКС ≈ 1.5 мин) — приемлемо.
- Тесты: geo **26 passed**.
