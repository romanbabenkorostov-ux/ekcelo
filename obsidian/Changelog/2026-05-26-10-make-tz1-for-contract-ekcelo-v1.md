# 2026-05-26 · 10_make_tz1_for_contract_ekcelo_v1

> Серия итераций v1 → v5. Финальная — v5 (этот файл).

## Контекст
Скрипт по аналогии с `09_make_reports_v1.py`: интерактивный CLI на Win10
(VSCode terminal, русская локаль), собирает в указанной папке проекта
выписки ЕГРН/ЕГРЮЛ/ЕГРИП (PDF/XML), ОСВ и реестры недвижимости (XLSX),
связывает активы с непосредственными правообладателями из ЕГРН и формирует
DOCX-документ «ТЗ-1» для договора на оказание услуг ЭКЦЕЛО.

## Что сделано
Файл `parser/scripts/pirushin_sosn_rocha_10_make_tz1_for_contract_ekcelo_v1.py`.

### Архитектура (1 файл, без новых модулей)
- **Console UI** (`ask_path` / `ask_yn`) — Win10 VSCode, ANSI-цвета,
  принимает Windows-пути с обратными слэшами и кавычками.
- **Accumulator** (`Asset`, `Beneficiary`) с дедупом по
  `cn::… / inv::… / own::…+addr`. Несколько правообладателей
  на актив — `holders: list[(name, inn)]` с дедупом по ИНН.
- **Фаза 1 (hybrid)**: чтение `<project>/_data/structure.json` /
  `enriched*.json` (выход 03/052), вытаскивание `cadastre_objects` +
  `beneficiaries`.
- **Фаза 2 (raw scan)**: `rglob`-обход, классификация по расширению
  (`.pdf` / `.xml` / `.xlsx`); пропуск служебных папок (`.git`,
  `.obsidian`, `_data`, `node_modules`, `__pycache__`, `.venv`).
- **Парсеры** (лёгкие, fail-safe):
  - PDF → `pdfplumber` (первые 8 страниц), классификация по маркерам;
  - XML → `ElementTree`, поиск тега `extract_about_property_*` или
    `egrul`/`egrip`;
  - XLSX → `openpyxl`, поиск шапки (наличие «наимен» + «кадастр»/«инв»/
    «адрес») и маппинг колонок (`name | address | cad | inv | owner |
    area | category | vri | purpose`).
- **Linker** `link_holders` — дообогащение `a.holders` по справочнику
  `acc.beneficiaries` (ИНН↔имя).

### Синтез колонки «Наименование актива»
4 реквизита склеиваются через `; ` (свойство `Asset.name`):
| Часть     | Что туда попадает |
|-----------|-------------------|
| `kind`    | земельный участок / здание / помещение / сооружение / машино-место / ОНС / оборудование |
| `param`   | площадь (кв.м), протяжённость (м), глубина (м), высота (м), объём (куб.м), степень готовности (%) |
| `purpose` | для **участка** → `Категория земель; ВРИ: …`; для прочих → «Назначение» |
| `own_name`| «Наименование» как самостоятельный реквизит выписки/ОСВ |

Мусорные подстроки служебной разметки PDF («Лист № …», «Всего листов
раздела …», «Всего разделов …», «не установлено», «отсутствует» и т.п.)
фильтруются единым словарём `_GARBAGE_SUBSTR` через `_clean(...)`.

### Колонка «Бенефициар»
- Префиксы вида `(правообладатели):`, нумерация `1.N`, хвостовой
  `(ИНН N)` срезаются (`_strip_holder_prefix`).
- Орг.-правовая форма сокращается (`_LEGAL_FORM_MAP`): «Общество с
  ограниченной ответственностью» → **ООО**, АО / ПАО / НАО / ЗАО / ОАО /
  ИП / ФГУП / ФГАУ / ФГБУ / ГУП / МУП / ГБУ / МБУ / АНО / ТСЖ / ТСН /
  ЖСК / СПК / ПК / КФХ. Длинные шаблоны матчатся раньше коротких
  («Публичное АО» не схлопнется в «АО»).
- Формат: `ООО "Х", ИНН N`. Несколько владельцев → склейка через `; `.

### Оформление DOCX
- A4, поля 2 см, Arial 10 pt.
- Шапка документа: 1-я строка — «Приложение 1» (курсив, по правому
  краю); перенос строки (НЕ новый параграф) → «к Договору
  №<YYYYMMDD>-1 от <ДД.ММ.ГГГГг.>» (текущая дата). Следом по центру —
  «Техническое задание №1» (обычный шрифт, без горизонтальной полосы:
  использован `add_paragraph`, а не `add_heading`, чтобы избежать border
  встроенного стиля Title).
- Таблица: «Light Grid Accent 1»; ширины колонок (см на A4):
  **№ = 1.0**, КН/Инв = 3.5, Наименование = 5.5, Адрес = 3.5,
  Бенефициар = 3.5. Прописываются через `w:tblGrid` + `cell.width`.
- Шапка таблицы: жирная, центрирование по горизонтали и вертикали,
  **повторяется на следующих страницах** (`w:tblHeader`).
- Колонка «№»: содержимое — по центру, явный `bold=False`
  (стиль таблицы навязывал жирный шрифт первой колонке).
- Блок после таблицы: «Подписи сторон:» (слева, обычный шрифт);
  2-колоночная таблица 8.5×8.5 см без рамок («Исполнитель:» /
  «Заказчик:» + строка `_________/_________/`). Границы удаляются
  через `w:tblBorders → nil` (хелпер `_remove_table_borders`).

## v6: многообъектные PDF-выписки

**Проблема**: из PDF-выписки ЕГРН с несколькими объектами (например,
несколько помещений в одном здании) парсилась только первая строка —
`scan_pdf_egrn` брал `next(iter(CAD_RE.findall(text)))`, теряя
объекты со 2-го по N-ный. Дополнительно — `_read_pdf_text(max_pages=8)`
обрезал большие выписки.

**Фикс**:
- `_read_pdf_text`: убран лимит `max_pages` (по умолчанию читаем все
  страницы; параметр оставлен для отладки).
- Добавлен `_split_text_by_cad(text)` — режет PDF-текст на блоки по
  сменам метки «Кадастровый номер: …»; повторы того же КН в шапке/
  подвале страниц игнорируются.
- `scan_pdf_egrn` теперь итерирует по блокам и для каждого вызывает
  общий помощник `_ingest_egrn_block` (он же подключён к `scan_xml`).

**Проверка тоже-же-проблемы в production-парсерах**:
- `parser/egrn_parser/parsers/pdf_parser.py::parse_egrn_pdf` —
  архитектурная установка «один PDF = один объект»; завязка на CLI /
  upsert / tests делает полный рефакторинг рисковым. Добавлено
  **детектирующее warning**: если в PDF замечено больше одного КН,
  логируется список и сообщается, что распарсен только первый.
  Дальнейший рефакторинг — отдельной задачей по запросу.
- `parser/egrn_parser/parsers/spravka_parser.py` — итерирует по
  строкам таблицы DOCX, бага нет.
- `parser/egrn_parser/parsers/xml_parser.py` — XML-выписка ЕГРН строго
  один объект (по корневому тегу), бага нет.

## v8: группировка по адресам + meta-строка

- **Шапка** документа: «Дата формирования: 2026-05-26 19:30:03» (жирный)
  заменена на «Уточнено 19:30:03 26.05.2026г.» (обычный шрифт). В meta
  передаются `time = "%H:%M:%S"` и `date = "%d.%m.%Yг."`.
- **Группировка строк таблицы**:
  - `_address_key(addr)` — берёт 2-3 последних компонента адреса
    («[город,] улица, дом») как ключ группы.
  - `_kind_order(kind)` — порядок типов внутри группы:
    1) земельный участок, 2) здание/сооружение/ОНС, 3) помещение/
    машино-место.
  - `_group_assets_for_table(assets)` — возвращает упорядоченные группы:
    адресные (alphabetical) → «— без адреса —» → «Оборудование»
    (последняя — БЕЗ внутреннего address-grouping'а, отсортирована по
    инв.№).
  - Каждая группа в DOCX начинается subheading-строкой с подписью
    (merge всех 5 колонок, bold, по левому краю).
- `_set_column_widths` теперь пропускает строки с merged-ячейками
  (subheading-row) — иначе Word плохо понимает width-инструкции для
  объединённой ячейки.

## v9: нормализация адресного ключа + защита от висячих строк

- **`_address_key`** переписан с last-3-components на покомпонентный
  разбор:
  - `_CITY_PREFIXES` — «г. / город / пгт / посёлок / с. / станица / …»;
  - `_STREET_PREFIXES` — «ул. / улица / пр. / пер. / бул. / шоссе / наб. / …»;
  - `_HOUSE_PREFIXES` — «д. / дом / зд. / стр. / влад. / …»;
  - `_DROP_PREFIXES` — «кв. / пом. / оф. / литер / корпус (как отдельная
    компонента)» — игнорируются.
  - `_normalize_house(s)` — «дом №60» / «д.60А» / «владение 7к1» →
    `д. 60` / `д. 60А` / `д. 7к1`.
  - Если ни улица, ни дом не распознаны — fallback к последним 2-м
    компонентам (без явных «дроповых»).
  - Результат: «ул. Суворова, дом №60, кв. 10» и «ул. Суворова, дом №60,
    кв. 10а» дают **один** ключ «ул. Суворова, д. 60»; адрес «Россия,
    Краснодарский край, г. Сочи, ул. Суворова, дом №60, кв. 13, 14»
    → «г. Сочи, ул. Суворова, д. 60».
- **Висячие строки**: добавлен `_set_row_keep_together(row, allow_split)`.
  - Для **subheading-row** — всегда `cantSplit` (короткая, не разрывается).
  - Для **data-row** — `cantSplit` по умолчанию; разрешаем разрыв
    `allow_split=True`, ЕСЛИ хотя бы одна ячейка превышает порог:
    «Наименование» > 544 chars (~30% A4 в колонке 5.5 см),
    «Адрес» / «Бенефициар» > 350 chars (~30% A4 в колонке 3.5 см).
    Этот порог соответствует ~16 строкам при Arial 10pt — иначе строка
    физически не поместится на странице и cantSplit вызвал бы клиппинг.

## v10: преамбула адреса + bare-house + MD-выгрузка

**Адрес-парсер**:
- `_strip_address_preamble(addr)` — срезает вводную фразу Россреестра
  «Местоположение установлено относительно ориентира, …. Почтовый адрес
  ориентира: …», оставляя только реальный адрес после неё.
- `_looks_like_bare_house(s)` — bare-numeric номер дома без префикса
  «д./дом» (например, «52», «60А», «1/3»). Применяется ТОЛЬКО если в
  адресе уже распознана улица; индексы (5+ цифр подряд) отсеиваются.
- Кейс заказчика: «Местоположение установлено… Почтовый адрес ориентира:
  Ростовская обл., г. Ростов-на-Дону, ул. Суворова, 52» теперь даёт
  ключ `г. Ростов-на-Дону, ул. Суворова, д. 52` (вместо v9 `…ул. Суворова`).

**MD-выгрузка параллельно с DOCX**:
- Добавлен `build_md(table1, out_path, meta)` — text-only, без
  зависимостей (если python-docx не установлен — MD всё равно соберётся).
- Структура зеркалит DOCX:
  - шапка «Приложение 1 к Договору …» (italic) → `# Техническое задание №1`;
  - meta «Уточнено …»;
  - `## Таблица 1. Список объектов для исследования` с одной MD-таблицей
    на группу (subheading-заголовок `### address` перед таблицей);
  - `## Подписи сторон:` с 2-колоночной таблицей-заглушкой.
- В `main()` сначала пишется `TZ1<ts>.md`, потом `TZ1<ts>.docx`; при
  отсутствии python-docx остаётся хотя бы MD.

  • «Приложение 1 к Договору №» → «Приложение 1 (обязательное)
    к Договору №»;
  • стиль таблицы «Light Grid Accent 1» → «Table Grid» (без банднинга и

    subheading-row;
  • subheading-row получает `keep_with_next = True` — заголовок группы
    не оставляется висячим в конце страницы (прикрепляется к первой
    data-row группы).
  Проблема: в реальной выписке (например, `Rostov_Socialisticheskaya.pdf`)
  «Наименование» занимает несколько строк («комнаты в подвале …; на 1
  этаже …; антресоли …»), а старый regex `[^\n]+` обрезал значение на
  первом переносе. Дополнительно v6-split ломал single-object PDF —
  в этом PDF метка «Кадастровый номер:» стоит в подвале страницы,
  ПОСЛЕ полей; split создавал блок, в котором полей нет.
  Решение:
  • Введён универсальный `_read_field(text, label_pat)` —
    мультистрочный экстрактор с lookahead до следующей известной
    метки из словаря `_FIELD_LABEL_PATTERNS` (50+ меток ЕГРН-выписки
    разделов 1 и 2 + page footer); внутренние переносы
    схлопываются в пробелы.
  • Все extractor'ы (`_extract_address` / `_extract_param` /
    `_extract_purpose` / `_extract_own_name`) переведены на новый
    парсер; для размерных параметров — двухступенчатый
    `_read_number = _read_field + первое число` (корректно даже
    при «Площадь, м2: 727.7» с единицей измерения в метке).
  • Откат `_split_text_by_cad` → `_distinct_cads(text)` (просто
    список уникальных КН). Если в тексте больше одного КН —
    логируем warning, но обрабатываем как одиночный объект.
  • Полная поддержка многообъектных PDF (с page-level grouping) —
    отдельной задачей.
  Smoke-тест на реальном PDF из Ростова-на-Дону: значение
  «Наименование» из 6 строк извлекается полностью (423 символа),
  совпадает с эталоном байт-в-байт.
