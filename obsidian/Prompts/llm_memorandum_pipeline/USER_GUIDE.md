# USER_GUIDE.md — Пошаговая инструкция для оператора

Привет! Это подробный туториал для тех, кто впервые делает меморандум через этот pipeline. Я веду «за руку», объясняю каждый шаг и проверочную точку.

Если ты опытный — открой `QUICKSTART.md` (короткий quickstart) и `INDEX.md` (точка входа).

## Зачем это нужно

В конце пайплайна у тебя на руках:

- **`final_report.md`** — инвестиционный меморандум по лоту (5 разделов + Преамбула).
- **`investment_slides.md`** — Marp-исходник презентации (5–6 слайдов).
- **`market_template.md`** — служебный шаблон рынка (накапливается между лотами).
- **`graph.html`** — интерактивный граф связей (если запускал парсер).
- **Фотоальбом** — если делал полевой осмотр.

Без pipeline сборка занимает 2–3 дня и легко плодит ошибки (разная актуальность выписок, забытые литеры, противоречия ЕГРН ↔ Поле). С pipeline — несколько часов, плюс темпоральная разметка фактов делает противоречия видимыми.

## Что вам потребуется

- **Python 3.11+** — чтобы запускать парсер (или коллегу из parser-team попроси).
- **claude.ai-аккаунт** (для ручного режима) ИЛИ **запущенный оркестратор** (после реализации по `orchestrator_spec.md`).
- **Доступ к репо `ekcelo`** — чтобы взять шаблоны из `templates/`.
- **Опционально: Marp CLI** — `npm install -g @marp-team/marp-cli`, чтобы рендерить презентацию в HTML/PDF.

✓ Проверьте: открывается ли claude.ai в браузере; есть ли `python --version` ≥ 3.11; репо `ekcelo` склонирован локально.

## Базовые понятия (глоссарий)

- **SSOT** (single source of truth, единый источник правды) — это файл `enrich_<lot_id>.json`. Все факты по лоту живут только в нём; меморандум и презентация рендерятся из SSOT.
- **Три слоя**:
  - **Юридический** — выписки ЕГРН (`enrich.egrn`).
  - **Градостроительный** — ГПЗУ, заключения Минкульта, ПЗЗ (`enrich.gpzu_minkult`).
  - **Физический** — полевой осмотр инженером (`enrich.field_inspection`).
- **Триада сценария** (`enrich.target_scenario`):
  - **was** — что было с лотом раньше.
  - **trigger** — что побудило запустить аудит.
  - **to_plan** — что планируется делать (аренда/ГАБ, редевелопмент, …).
  - От `to_plan` зависит, какие риски «спящие», а какие «критические».
- **Темпоральный контракт**:
  - `evidence_level=1` — `registered_date` (когда зарегистрировано в реестре, приоритет).
  - `evidence_level=2` — `document_date` (когда сформирован документ выписки).
  - Конфликт: свежее побеждает старое; при равных датах — registered побеждает document_date.
- **`<SYSTEM_MARKET_TEMPLATE>`** — служебный блок в ответе модели, в котором живёт обновлённый шаблон рынка. Не показывай его клиенту. Pipeline/оркестратор автоматически вырезает его и кладёт в `market_template.md`.

✓ Проверьте: можешь объяснить эти 5 терминов своими словами.

## Шаг 1 — Подготовить парсерный JSON

Зайди в репо `ekcelo`, в корне выполни:

```bash
python -m egrn_parser export --json out.json
```

Если parser-команда у вас в проекте называется иначе — спроси команду парсера. На выходе у тебя `out.json` — это структурированный ЕГРН-слой.

✓ Проверьте: `out.json` открывается, в нём есть `tables.objects[]` со списком объектов и КН.

## Шаг 2 — Подготовить граф связей

```bash
python parser/scripts/04_nspd_graph_v14.py --output <Название_проекта>/graph.html
```

Граф появится в корне твоего проекта. Pipeline найдёт его рекурсивно.

✓ Проверьте: `graph.html` открывается в браузере (если CORS блокирует локальный file://, запусти `python -m http.server 8000` из папки с `graph.html` и открой `http://localhost:8000/graph.html`).

## Шаг 3 — Заполнить YAML-карточки

Скопируй из `obsidian/Prompts/llm_memorandum_pipeline/templates/` в свою рабочую папку лота:

- `target_scenario.yaml.tpl` → заполни триаду (was / trigger / to_plan). **Обязательно.**
- `documents_dates.yaml.tpl` → для каждой выписки ЕГРН/ЕГРЮЛ/ЕГРИП укажи `registered_date` и/или `document_date`. **Обязательно.**
- `gpzu_minkult.yaml.tpl` — если есть ГПЗУ/Минкульт. Опционально.
- `field_inspection.yaml.tpl` — если был полевой осмотр. Опционально.
- `photo_album_index.yaml.tpl` — если есть фотоальбом. Опционально.

Если данных по слою нет — не заполняй файл, просто скажи Этапу 1 «нет данных». Слой попадёт в `missing_layers[]`.

✓ Проверьте: `target_scenario.yaml` имеет все три ненулевых строки; `documents_dates.yaml` имеет хотя бы одну запись с хотя бы одной из двух дат.

## Шаг 4 — Собрать `enrich_<lot_id>.json`

**Путь А (новичку, через claude.ai):**

1. Открой `01_intake_and_pipeline.md`, скопируй блок «СИСТЕМНЫЙ ПРОМПТ» в системное сообщение claude.ai.
2. Скопируй блок «ПОЛЬЗОВАТЕЛЬСКИЙ ПРОМПТ», подставь:
   - `lot_id` — короткое имя лота (без пробелов, например `pirushin_001`).
   - `PARSER_JSON` — содержимое `out.json` из Шага 1.
   - YAML-карточки из Шага 3.
3. Модель проведёт тебя через 4 фазы: pre-flight → три вопроса target_scenario → темпоральный приёмник → доп.слои.
4. На выходе — code-block с JSON. Сохрани его как `<Название_проекта>/Memorandum/_data/enrich_<lot_id>.json`.

**Путь Б (через оркестратор):**

`POST /lots/<lot_id>/run` с body `{"workspace_path": "<...>"}`. Оркестратор сам спросит target_scenario через web-форму и соберёт SSOT.

✓ Проверьте: `enrich_<lot_id>.json` валиден (JSON.parse не падает), есть `target_scenario`, `documents_dates[]`, `facts_index[]`. Если есть `conflicts[]` — это нормально, Этап 2 их разрешит.

## Шаг 5 — Получить меморандум и презентацию

**Путь А (через claude.ai):**

1. Открой `02_memorandum_prompt.md`, скопируй системный промпт в системное сообщение. Скопируй пользовательский, подставь:
   - `ENRICH_JSON` — содержимое SSOT.
   - `MARKET_ANALYSIS` — сырой текст анализа рынка по ФСО №7.
   - `EXISTING_MARKET_TEMPLATE` — пусто на первом прогоне; со второго лота — содержимое `Memorandum/market_template.md` с прошлого.
   - `graph_status` — `TRUE` если есть `graph.html`, иначе `FALSE`.
2. Модель выдаст единый ответ из трёх блоков:

   ```
   <SYSTEM_MARKET_TEMPLATE>
   ...
   </SYSTEM_MARKET_TEMPLATE>

   ## Преамбула: сравнительный анализ матрицы актива
   ...
   ## 5. Инфраструктурный капитал и карта рисков
   ...

   <!-- MARP_START -->

   ---
   marp: true
   ...
   ```

3. Разрежь руками:
   - Всё внутри `<SYSTEM_MARKET_TEMPLATE>...</SYSTEM_MARKET_TEMPLATE>` → `Memorandum/market_template.md` (без тегов).
   - Всё между концом `</SYSTEM_MARKET_TEMPLATE>` и `<!-- MARP_START -->` → `Memorandum/final_report.md`.
   - Всё после `<!-- MARP_START -->` → `Memorandum/investment_slides.md`.

**Путь Б (через оркестратор):**

После Шага 4 оркестратор сам пройдёт Фазы 2–4 и положит три файла в `Memorandum/`.

✓ Проверьте: в `final_report.md` нет тегов `<SYSTEM_…>`; в `investment_slides.md` начинается с `---\nmarp: true`; в `market_template.md` есть 4 заголовка категорий.

## Шаг 6 — Отрендерить презентацию

Из `Memorandum/`:

```bash
marp investment_slides.md -o investment_slides.html
marp investment_slides.md -o investment_slides.pdf --pdf
```

Если `marp` не установлен:

```bash
npm install -g @marp-team/marp-cli
```

Если нет Node.js — можно открыть `investment_slides.md` через VS Code с расширением «Marp for VS Code» и экспортировать оттуда.

✓ Проверьте: `investment_slides.html` открывается в браузере, бейджи `badge-risk`/`badge-asset` отрисованы цветными.

## Шаг 7 — Собрать комплект для заказчика

В папке `Memorandum/` у тебя:

- `final_report.md` — меморандум (отправь заказчику как `.md` или сконвертируй в PDF через любой md→pdf).
- `investment_slides.html` (или `.pdf`) — презентация.
- `../graph.html` — интерактивный граф (приложи отдельно).
- Фотоальбом (если делал) — отдельный архив.

`market_template.md` и `_run_log.jsonl` **не отправляй** заказчику — это служебные файлы.

✓ Проверьте: открой `final_report.md` глазами, убедись что Преамбула содержит дату актуальности SSOT и упоминание разрешённых конфликтов.

## Частые проблемы и решения

- **«Модель не выдала блок `<SYSTEM_MARKET_TEMPLATE>` — что делать?»** → Проверь, что в системном промпте Этапа 2 вставлен полный текст `market_injector_prompt_block.md` (раздел «ЭТАП 0»). В новой сессии claude.ai системный промпт мог потеряться — вставь его заново.
- **«Две выписки на разные даты противоречат друг другу — как решается?»** → Pipeline применит правило конфликтов: свежее побеждает; при равных датах registered побеждает document_date. Проверь `conflicts[]` в SSOT — там видно, какой факт «победил».
- **«Граф `graph.html` не открывается локально (file://)»** → Браузеры блокируют JS из локальных файлов. Запусти `python -m http.server 8000` из папки с графом и открой `http://localhost:8000/graph.html`.
- **«`marp` команда не найдена»** → `npm install -g @marp-team/marp-cli`. Если нет npm — поставь Node.js (`https://nodejs.org`) или используй VS Code-расширение.
- **«target_scenario неизвестен — что писать в `to_plan`?»** → Если у заказчика нет решения, поставь «текущее использование без изменений (аренда/ГАБ)» — это самый частый дефолт. Меморандум классифицирует ОКН/археологию как СПЯЩИЕ риски. Когда решение появится — перегенерируй меморандум с новым `to_plan`.
- **«Парсерный JSON большой, не влезает в claude.ai»** → Используй оркестратор (он шлёт через API). Или разрезай парсерный JSON по объектам и собирай SSOT в несколько подходов.
