# EKCELO — описание системы (сведено по C2-схеме)

> Сведение пяти точек зрения (экономист/финансист/бухгалтер/юрист, системный
> архитектор, технолог, консультант, специалист по БД) в одно непротиворечивое
> описание. Дубли убраны, формулировки приведены к утверждённой схеме данных C2
> (`contracts/db/SCHEMA_SPEC.md`, `models.py`, `GRAPH_DB_PRINCIPLES.md`,
> `DOC_CLASSIFIER_SPEC.md`, `PARSER_VOCAB_MAP.md`). Версия: 1.0 · 2026-06-04.
>
> Изменения относительно исходного черновика собраны в Приложении A.

---

## 1. Архитектурная парадигма

EKCELO — это не жёсткая учётная модель ERP, а **граф знаний (Knowledge Graph)** с
**вероятностной истиной**. Система не хранит «факт» как абсолют: она хранит
**Утверждение (Assertion)**, подкреплённое **Доказательствами (Evidence)** разной
жёсткости из разных **Источников (DataSource)**, и динамически считает по ним
**индекс уверенности** `confidence_score ∈ [0..1]`.

Граф — **логический поверх табличной БД**, а не отдельный движок. Узлы и рёбра —
это таблицы (`entities`, `relations`); их рендерят `graph.html` (контракт C1
`graph_node_id`) и ViewModel (C4). Обход цепочек — рекурсивные SQL-CTE (PostgreSQL)
/ итеративно (SQLite). Neo4j/Memgraph **не вводится**. Геометрия в граф не кладётся —
только `bbox` по ссылке (см. §7).

**Базовая триада графа:**

```
Entity  →  Relation (+domain)  →  Assertion  →  Evidence
узел       ребро модели мира      утверждение    доказательство (source + weight)
```

- **Relation ≠ Assertion.** `Relation` — связь в модели мира; `Assertion` —
  утверждение о её существовании с `confidence`; `Evidence` — доказательство
  (документ/источник). Это снимает перегрузку единой таблицы рёбер.
- **Узел = `entities`** — адресная строка над таблицей-владельцем. Любая сущность
  (объект, субъект, документ, КИП, лот, БА, точка разграничения) может быть узлом.
- **Ребро = `relations`** + 1:1 доменное расширение (`legal_/tech_/spatial_/accounting_relation`).

## 2. Многослойные миры

Система растёт от физического ядра процессов к финансово-административной оболочке:

1. **Мир производства и материальных потоков (ядро).** Технологический слой:
   движение сырья, полуфабрикатов, готовой и побочной продукции через инфраструктуру
   и КИП. Моделируется направленным графом потоков (домен `tech`).
2. **Мир объектов.** Слой активов: недвижимость, движимые ОС, НМА. Изучается парсингом
   и осмотрами, представляется сюрвей-презентацией.
3. **Мир лотов.** Коммерческий слой: лот как набор объектов под сделку (домен `commercial`).
4. **Мир заказов.** Операционный слой: договор, сопровождение сюрвей-презой,
   взаимодействие бенефициаров и асесоров.
5. **Мир админа (оболочка).** Биллинг, вознаграждение за сюрвей-презу, поддержание
   парсинга и вьюера.

**Два графовых сегмента** (срезы одного логического графа, не разные БД):
- **Юридический граф** — бенефициары, документы-основания, объекты, права/обременения,
  битемпоральность (домены `legal`, частично `commercial`).
- **Технологический граф** — инфраструктура, КИП, материальные потоки (домены `tech`,
  `spatial`, `accounting`).

## 3. Объект и шесть степеней свободы

Любой объект учёта описывается шестью измерениями. Они же — маппинг на 4 фасета
ViewModel (C4) плюс классификация и время:

| # | Степень свободы | Вопрос | Где в схеме |
|---|-----------------|--------|-------------|
| 1 | Пространство | **где это** | `geometries` (WKT/GeoJSON, srid) → фасет `geo` |
| 2 | Классификация | **что это** | `objects.object_type` / `devices` / НМА → `physical` |
| 3 | Субъект права | **чьё это** | `relations[legal]` + `subjects` → `ownership` |
| 4 | Физика и состояние | **какое это** | `objects` + `object_etp_profile` → `physical` |
| 5 | Регламенты использования | **как использовать** | разрешённые виды / ограничения (legal/restriction) |
| 6 | Время | **когда это** | битемпоральные поля → `temporal` |

«Понятие основного средства относится ко всем объектам.» Абстракция **Asset**
ложится на схему так:
- `RealEstate` (недвижимость: ЗУ + ОКС) → таблица `objects` (единая; `object_type ∈
  {land, building, room, structure, ons}`);
- `FixedAsset` (движимое ОС: оборудование, КИП, транспорт) → `devices` и
  `entities(kind ∈ {equipment, device, accessory})`;
- `IntangibleAsset` (НМА) → компонентная БД `nma.db` (ФИПС, ADR-P03), мост через
  `subject_external_ref`; к ОС не относится, бенефициар — ЮЛ/ИП.

### 3.1 ГИС-топология недвижимости

- ЗУ и ОКС связаны **многие-ко-многим** (`relations[spatial]`: `CONTAINS`/`INSIDE`):
  ОКС может стоять на нескольких ЗУ, на одном ЗУ — несколько ОКС.
- Иерархия вложенности: ЗУ ⊃ здание ⊃ помещение (рёбра `CONTAINS`). `accessory`
  (принадлежность) — узел `entities(kind='accessory')`, ребро `CONTAINS` к родителю,
  гео-точка в `geometries(geometry_type='POINT')`.
- ОКС: здания (ЗД), сооружения (СООР), помещения (ПОМ), ОНС. Могут быть многоконтурными,
  со сквозными инвентарными/условными/кадастровыми номерами.
- Контуры — из техпланов, техпаспортов, выписок ЕГРН.

## 4. Субъекты (бенефициары и роли)

**Тип субъекта** (`subjects.subject_type`, Enum) — только правовая форма:
`INDIVIDUAL` (ФЛ), `LEGAL_ENTITY` (ЮЛ), `INDIVIDUAL_ENTREPRENEUR` (ИП),
`STATE_BODY` (госорган/муниципалитет).

> **Важно:** бенефициар, асесор, админ — это **роли** (контракт C6), а **не** типы
> субъекта. Один и тот же субъект может быть бенефициаром в одном заказе и контрагентом
> в другом. Роль не зашивается в `subject_type`.

- **ЮЛ:** ИНН (10), ОГРН (13), КПП. Единственный КПП — нет филиалов; доп. КПП →
  обособленные подразделения (`subject_kpp`, множественный). Один ИНН может менять
  наименование во времени (`subject_names` с `valid_from/valid_to`).
- **ИП:** ИНН (12, совпадает с ИНН физлица). Статус ИП (ОГРНИП) может многократно
  присваиваться/аннулироваться → `ip_status_periods` (повторяемые периоды).
- **Руководитель.** Лицо из раздела ЕГРЮЛ «Сведения о лице, имеющем право без
  доверенности действовать от имени ЮЛ» — это **отдельный субъект** (ФЛ **или** ЮЛ —
  управляющая компания), связанный ребром `MANAGES` → ЮЛ. Руководитель — не тип
  субъекта и не учредитель: он управляет, но может не владеть долей.
- **Банковские реквизиты:** у ЮЛ/ИП/ФЛ — массив счетов (`bank_accounts`): банк, БИК (9),
  коррсчёт (20), расчётный счёт (20), дата открытия, дата закрытия (`null` по умолчанию).
- **Внешние компонентные БД (ADR-P03):** `innogrn.db` (checko: руководители,
  учредители, ОКВЭД, режим налогообложения) и `nma.db` (ФИПС). Связь — `subject_external_ref`
  по ИНН/ОГРН. Не уплощаются в основную схему.

### 4.1 Корпоративные связи (домен `legal`, категория `corporate`)

Связи ЮЛ→субъект моделируются рёбрами категории **`corporate`** — отдельно от вещных
прав на недвижимость:

| Код ребра | Смысл | Направление (from → to) |
|-----------|-------|-------------------------|
| `FOUNDER_OF` | учредитель/участник (владение долей УК) | учредитель → ЮЛ |
| `MANAGES` | руководитель/УК (действует без доверенности) | руководитель → ЮЛ |
| `CONTROLS` | бенефициарный контроль (доля в `meta`) | контролирующий → ЮЛ |
| `BRANCH_OF` | обособленное подразделение | подразделение → ЮЛ |
| `SUCCESSOR_OF` | правопреемство (реорганизация завершена) | правопреемник → правопредшественник |
| `REORGANIZING_WITH` | участие в незавершённой реорганизации | ЮЛ ↔ ЮЛ (tentative) |

### 4.2 Реорганизация и правопреемство

**Правопреемство** возникает при реорганизации ЮЛ: компания меняет форму, но (в отличие
от ликвидации) права и обязанности не исчезают, а **переходят к правопреемнику** —
включая долги, налоговые обязательства, обязательства по штату и имуществу. Источник —
разделы ЕГРЮЛ «Сведения о правопреемнике» / «Сведения о правопредшественнике»; основание
ФЗ-129 ст. 5 п. 1 пп «ж».

**Виды реорганизации** (`relations.meta.reorg_type` на ребре `SUCCESSOR_OF`):

| `reorg_type` | Рус. | Структура |
|--------------|------|-----------|
| `merger` | слияние | A + B → C (новое ЮЛ, A и B прекращаются) |
| `affiliation` | присоединение | B → A (B прекращается, права к A) |
| `division` | разделение | A → B + C (A прекращается) |
| `spin_off` | выделение | A → A + B (A продолжает, B новое) |
| `transformation` | преобразование | A → A′ (смена ОПФ) |

Эти структуры выражаются **набором рёбер** `SUCCESSOR_OF` (правопреемник →
правопредшественник): разделение/слияние = несколько рёбер с общим `meta.reorg_type`
(+ опц. `meta.reorg_event_id` для группировки одного события).

> **Правопреемник ≠ учредитель.** Правопреемник наследует права и обязанности
> (ребро `SUCCESSOR_OF`); учредитель создаёт ЮЛ и владеет долей (ребро `FOUNDER_OF`).
> Сам факт правопреемства новых учредителей автоматически не создаёт — состав
> участников меняется только если это отдельно зафиксировано в выписке.

**Незавершённая реорганизация.** Пока процесс не завершён, правопреемник прямо в
выписке не указан — есть только раздел «Сведения о юрлицах, участвующих в реорганизации».
Это моделируется ребром `REORGANIZING_WITH` (tentative, низкий `confidence`), которое при
завершении реорганизации сменяется на `SUCCESSOR_OF` (старое ребро `superseded_at`).

## 5. Юридический сегмент: права, обременения, документы

Право — это **ребро** (`relations[legal]`), а не отдельная сущность-узел. Атрибуты
права лежат в `legal_relation` (рег. номер, документ-основание, код права). Не
смешиваем вещные права и ограничения на уровне типа ребра:

| Категория | Коды (`relation_types.code`) |
|-----------|------------------------------|
| right | `OWNS`, `LEASES`, `OPERATES`, `SERVITUDE`, `GRATUITOUS_USE` |
| encumbrance | `MORTGAGED_BY` (залог), `ARRESTED_BY` (арест) |
| restriction | `RESTRICTED_BY` (СЗЗ, водоохранные зоны, ОКН) |
| corporate | `CONTROLS`, `FOUNDER_OF`, `MANAGES`, `BRANCH_OF` |

**Относительность прав.** То, что для одного бенефициара ограничение, для другого —
бенефит: на один ЗУ возможны одновременно `OWNS` (собственник), `LEASES` (арендатор)
и `SERVITUDE` (проход/проезд по решению суда или договору) — три валидных ребра.

**Документы-основания** (`documents` + узел `entities(kind='doc')`): акт ввода в
эксплуатацию, решение суда, акт кадастрового инженера, ДКП, свидетельство,
приватизация. Возникновение/переход/прекращение права фиксируется ребром
`ESTABLISHES` (`doc_links.relation_code`). Цепочка документов прослеживает историю
прав от первичного возникновения.

**Жизненный цикл недвижимости:**
- Признаётся недвижимостью по акту ввода в эксплуатацию / решению суда / акту
  кадастрового инженера; статус возникает при госрегистрации с присвоением
  кадастрового (ранее условного) номера.
- Перестаёт быть недвижимостью с момента госрегистрации на основании акта обследования
  кадастрового инженера; ОКС — при зафиксированном уничтожении.
- Ремонт сохраняет ранее задокументированные физ. параметры; реконструкция может их менять.

Источники прав/бенефициаров: выписки ЕГРН, свидетельства, договоры; описание
субъектов и связей — ЕГРЮЛ/ЕГРИП.

## 6. Бухгалтерско-правовой учёт (домен `accounting`)

Права на ОС верифицируются по **ОСВ** бенефициара (бенефициар — по имени файла ОСВ):
собственность — счёт `01.01`, аренда/лизинг — `01.03` и `01.К`. Это **отдельный
домен** `accounting` (рёбра `ON_BALANCE_OF`, `LEASED_IN_BALANCE`, `CAPITALIZED_BY`),
**не смешиваемый** с legal-`OWNS`/`LEASES`: балансодержатель может ≠ юр. собственник
(оба ребра валидны).

**Классификация хозяйственных операций:**
- **Активно-активные:** купля/продажа/мена прав между независимыми бенефициарами за
  деньги (трансформация структуры активов); перевод ОС в оборотные запасы (Дт 10 — Кт 01.01,
  «разобрать здание в кирпич») и ввод в эксплуатацию (Дт 01.01 — Кт 08).
- **Пассивно-пассивные:** передача активов между материнской и дочерней структурами в
  обмен на изменение обязательств / взнос в УК по справедливой стоимости.
- **Решения уполномоченных органов:** наложение/изменение/снятие прав, обременений,
  ограничений, обеспечительных мер; признание и восстановление нарушенного права.

Фиксация сделки: по рыночной стоимости при win-win без паритетного ущемления; по
оценочной (справедливой) стоимости при внутригрупповых операциях; решением органа —
при принудительном изменении прав.

## 7. Геометрия

`geometries`: `geometry_type ∈ {POINT, LINESTRING, POLYGON, MULTIPOLYGON}`,
`coordinates_wkt` (WKT), `geojson`, `bbox`, `original_srid` (исходная СК выписки,
напр. **МСК-61**), `srid` (по умолчанию **4326 / WGS-84** — обязательно для KMZ, C1).

- Перевод **МСК-61 → WGS-84** обязателен для экспорта в `.kmz`.
- Хранение переносимо: WKT/GeoJSON + `srid`. **PostGIS — опционально и только на
  PostgreSQL** (не тип данных по умолчанию; SQLite работает на WKT/JSON1).
- В графе геометрии нет — только `geometries.bbox` для быстрого поиска (правило
  разделения доменов).
- Поворотные точки контура примерно соответствуют GPS; недвижимость может быть и без контура.

## 8. Технологический сегмент: инфраструктура, КИП, потоки

Физическое перемещение (семечка → автовесы → силосы → маслопрессовый цех → массовый
расходомер → жд-терминал; побочное: лузга/шрот → весы → отгрузка) — направленный граф,
домен `tech`. Инфраструктура (цеха, склады, весовые, жд-пути) — узлы-объекты; потоки —
рёбра (`MOVED_TO`, `FEEDS`, `TRANSFORMS_TO`, `CONNECTED_TO`).

**Типы технологических узлов:**
1. **Проточный узел** (автовесы, расходомер) — масса проходит транзитом, остаток = 0.
2. **Узел-накопитель** (силос, маслобак, склад тары) — `devices.max_capacity` хранится;
   **`current_level` НЕ хранится — вычисляется из `flow_events`** (переходящий остаток
   на контрольную дату).
3. **Поток-трансформатор** (термопластавтомат, пресс) — меняет физ. сущность
   (ПЭТ-крошка → бутылка) или делит на фракции (семечка → масло + шрот + лузга).
   Параметры на `tech_relation`: `material_type_in/out`, `conversion_ratio`, `loss_factor`.

**Компоненты КИП** (датчики, расходомеры, весовые терминалы, трекеры) — самостоятельные
сущности `devices` (ОС). Иерархическая привязка: к гео-точке; к объекту недвижимости
(датчик уровня в Силосе №3); к технологическому процессу (ребру потока).

**Разделение слоёв данных КИП:**
- `Device` — физический прибор (хранится).
- **Telemetry — сырой поток — ВНЕ проекта** (Time-Series БД); с графом напрямую не
  джойнится.
- `FlowEvent` — нормализованное бизнес-событие в графе: `relation_id` (маршрут),
  `device_id` (или `document_id` для ручного ввода/инвентаризации), `timestamp`,
  `quantity`, `unit`, `event_type`:
  - `DISCRETE` — одно взвешивание (машина/вагон);
  - `CONTINUOUS` — расходомер за смену/период (одна агрегатная запись);
  - `CORRECTION` — усушка, утруска, влагопоглощение, результат инвентаризации.

Динамику (тонны/кубы/штуки) пишем **только** в `flow_events`, не в рёбра — иначе
ломается битемпоральность и структура ребра.

**Коммуникации** — линейные сооружения; имеют узел **точки разграничения балансовой
принадлежности** (`entities(kind='demarcation_point')`). Узел разрывает ребро
`CONNECTED_TO` между сетевой организацией (бенефициар А) и объектом (бенефициар Б),
разграничивая зоны эксплуатационной ответственности. Импорт — из договоров снабжения.

## 9. Коммерческий слой: лот, заказ, договор, УПД

**Лот формируется с двух концов** (объект и бенефициар), наполняется/сужается по
групповым и индивидуальным степеням свободы; на этапе КП лот «живой» (`lots` +
`as_of_date`, `include_json`, `exclude_json`).

**Замораживание.** При переходе лота в «Договор подписан» / «Выставлен счёт» система
делает неизменяемый слепок состава — `lot_snapshots(frozen_data, reason, xsd_version)`.
**Все печатные формы, XML УПД и сюрвей-презентации генерируются строго из
`LotSnapshot`, а не из живого лота** — иначе теряются условия, под которые подписан
договор и сформирован УПД.

**Заказ** (`orders`) не имеет жёсткой последовательности наполнения. **Договор**
(`contracts`): тело, реквизиты сторон (исполнитель/заказчик), ТЗ (`tz_json`), состав
лота (`lot_snapshot_id`). По выполнении — **УПД** (`upd_documents`), часто XML,
проверяется по XSD (следить за `xsd_version`).

**Налоги и УПД.** Парсинг контрагентов через checko.ru / dadata.ru с определением
режима. В профиле субъекта: `vat_mode ∈ {OSNO, USN, USN_VAT}`, `vat_rate ∈
{0, 5, 7, 10, 20}`, `vat_exemption_reason` (ст. 149 НК РФ). Правило генератора:
- ОСНО → УПД **статус 1** (акт + счёт-фактура, НДС выделен);
- УСН → УПД **статус 2** (чистый акт без НДС);
- **`USN_VAT` ⇒ статус 1** — несмотря на упрощёнку (реформа УСН-2025: доход > 60 млн ₽
  → НДС 5/7/20). `upd_documents` хранит `status`, `xsd_version`, `validated`.

**Бизнес-актив (БА)** — высший уровень группировки (`entities(kind='business_asset')`,
рёбра `GROUPS`): формируется по волевому решению, адресу, набору ОС или НМА. Субъект
мира лотов получает скомпонованное представление БА/лота в `.kmz`.

## 10. Провенанс: источники, веса, уверенность

Любая связь субъект↔объект — `Assertion`; к одному `Assertion` привязано ≥1 `Evidence`;
у каждого Evidence — `source_type` и `weight`.

**Шкала весов** (`SOURCE_WEIGHTS`):

```
EGRN = 1.0   COURT_DECISION = 1.0   OSV = 0.8   NSPD = 0.6
EXIF = 0.5   LLM = 0.4   SURVEY_MANUAL = 0.3
```

> **EGRUL / ЕГРИП НЕ доказывают `OWNS` объекта** — они питают атрибуты субъекта и
> корпоративные рёбра (`FOUNDER_OF`/`MANAGES`). Это отличие от наивного «любой
> источник = право».

`confidence_score = 1 − Π(1 − wᵢ)` по согласным доказательствам (минус штраф за
конфликт). Конкурирующие утверждения хранятся все; активное = `max(confidence)`,
tie-break по `recorded_at`; ручной override асесора помечается (`asserted_by`).

Документы во времени меняют признаки объектов, субъектов и связей — это фиксируется
новыми `Assertion`/`Evidence` поверх старых, без удаления истории.

## 11. Документы и классификатор

Каждый документ рассматривается как «**какие параметры каких сущностей он обогащает**»
(`doc_links`: `документ → target_table.target_field`, `relation_code ∈ {ESTABLISHES,
EVIDENCES, DEPICTS}`). Парсер-классификатор (`DOC_CLASSIFIER_SPEC`) на вход берёт файл
или рекурсивно обходит папку: detect → classify → extract → anchor → route → provenance.
Якорь подтверждается оператором один раз на группу (папку), не на файл.

**Базы по концентрации документов** (стартовый набор):
- ЕГРН (выписки PDF+XML) → §1–§5 `objects`/`rights`/`extracts`/`restrictions` + `geometries`;
- ЕГРЮЛ + ЕГРИП → `subjects`/`subject_kpp`/`subject_names`/`ip_status_periods` (+ innogrn.db);
- НМА → `nma.db` (ФИПС, ADR-P03);
- фото/фотофиксация (EXIF) → `geometries(POINT)` + разметка `object_etp_profile` (LLM);
- объекты ОС + связи с бенефициарами (степени свободы) → граф §7;
- договоры мира админа (стороны, банк. реквизиты, тело, услуги, оплата), счета, УПД (XML) → §11.

## 12. Битемпоральность (когда это)

Все изменяемые таблицы фактов и рёбра/утверждения несут две оси времени:
- **реальное время:** `valid_from` / `valid_to` (в графе — `Since` / `Until`);
- **системное время:** `recorded_at` / `superseded_at` (в графе — `KnownSince` / `KnownUntil`).

Справочники — без битемпоральности. Срез графа «на дату D»:
`valid_from ≤ D < valid_to AND superseded_at IS NULL`. Лот тянет срез по `as_of_date`.
