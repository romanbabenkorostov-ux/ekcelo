# Viewer: слои core/adapters/ui

Справочник по архитектуре, введённой ADR-003.

## Зачем

Подготовка к React-порту без необходимости переписывать логику с нуля. После рефакторинга порт = замена слоя `ui/` на React-компоненты; `core/` и `adapters/` идут как есть.

## Слои

### `viewer/core/` — чистые функции

**Правила:**
- Принимает данные → возвращает данные. Никаких побочных эффектов.
- Запрещены глобалы: `document`, `window`, `navigator`, `location`, `localStorage`, `sessionStorage`, `fetch`, `XMLHttpRequest`, `alert`/`confirm`/`prompt`. Ловится ESLint-правилом `no-restricted-globals` (см. корневой `.eslintrc.json` (секция `overrides` для `viewer/core/**/*.js`)).
- Разрешено: `globalThis.crypto` (Web Crypto API доступно и в браузере, и в Node 20+).
- Каждая публичная функция покрыта юнит-тестом в `tests/viewer/`.

**Что туда уходит:** хэширование, escape (HTML/XML), форматтеры (координаты, расстояния), конверторы (CRS, единицы), сборщики строк (URL, KML, CSV), фильтры/агрегации/группировки списков объектов, иммутабельные структуры состояния.

### `viewer/adapters/` — мост к внешнему миру

**Правила:**
- Тонкие функции «прочитать из мира → отдать в core», «взять из core → положить в мир».
- Никакой бизнес-логики. Никакого UI.
- Может использовать `fetch`, `localStorage`, `File System Access API`, `crypto.subtle`.

**Что туда уходит:** клиент REST-API, обёртки над storage, File System Access, EXIF-чтение, импорт KML/XLSX (только парсинг файла в структуру для core).

### `viewer/ui/` — тонкая DOM-обёртка

**Правила:**
- Импортирует core, подписывается на изменения состояния, рендерит.
- Обработчики кликов — здесь.
- **Бизнес-логика и форматирование данных — НЕ здесь.** Если нужно вычислить — вызвать core-функцию.

**Что туда уходит:** Leaflet-обёртка, дропдауны, тосты, кнопки, инициализация карты, exif-write UI, drag-handlers.

## Состояние

Единый store с подпиской (паттерн observable). На vanilla — 50 строк. Когда поедем на React — store заменяется на Zustand/Context **один к одному**, потому что API совместимо.

Никаких `window.__ekceloRole` — всё через store. Глобал-присваивания терпимы только как **временный мост** при выносе функций из `index.html` (когда классический `<script>` зовёт перевыделенную функцию по старому имени), и убираются, как только последний вызывающий тоже переедет в модули.

## Сборщик

Не нужен. ES-модули нативно через `<script type=module>` в `index.html`. Пути — относительные, файлы `.js` отдаются GitHub Pages как `text/javascript` без преобразований.

Когда понадобится React — введём Vite одновременно с миграцией `ui/` на JSX, в отдельном PR.

## Тесты

- `tests/viewer/*.test.mjs` — формат `.mjs`, через встроенный `node:test`.
- Запуск локально: `node --test tests/viewer/*.test.mjs`.
- В CI: `.github/workflows/tests.yml` на каждом PR с правками `viewer/**`.
- Никаких npm-зависимостей.

## Acceptance для нового кода

| Что | Где |
|---|---|
| Чистая функция (форматтер, конвертор, фильтр) | `viewer/core/` + тест |
| fetch / storage / File System | `viewer/adapters/` |
| DOM-обработчик, Leaflet-связка, рендер | `viewer/ui/` |
| Inline `<script>` в `viewer/*.html` (кроме `v2961.html`) | **запрещено** для новой логики |
