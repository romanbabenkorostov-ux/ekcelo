# Золотой путь Ekcelo — для экономиста

> Этот документ — «история одного лота» от ваших исходников до готового пакета
> для бэкенда. Показывает: **какие задачи ставятся, как решаются, какие проблемы
> по дороге возникали и как мы их закрыли**. К нему приложены **мок-данные** (можно
> запустить демо и увидеть весь путь вживую).

## Зачем это
У вас на руках разрозненное: выписки ЕГРН, перечни виноградников из залога,
техкарты, ОСВ (техника), фото, ваши экспертные правки. Ekcelo собирает это в
**единый оценочный пакет (Bundle)** по объекту/лоту: земля + насаждения + уход +
погода + техника + ваши правки → один `db.sqlite` + карта (KMZ) + manifest.

## Золотой путь — 8 шагов (задача → решение → проблема → как закрыли)

### Шаг 1. Земля: контуры участка
**Задача:** определить тип участка (обычный ЗУ / единое землепользование ЕЗП /
многоконтурный МКУ) и записать контуры.
**Решение:** геометрия из NSPD → классификация + контуры в БД.
**Проблема, которая возникала:** у ЕЗП геометрия — это тоже «много контуров», и
система ошибочно понижала ЕЗП до МКУ. → **Закрыто:** ЕЗП определяется по дочерним
КН/маркеру выписки и геометрией НЕ понижается.

### Шаг 2. Виноградники: перечень насаждений
**Задача:** превратить текст залогового перечня в структуру (сорт-привой, подвой,
год высадки, площадь, кусты, № в федеральном реестре).
**Решение:** парсер `vineyard_perechen` разбирает блоки «Многолетние насаждения…
Предмет залога N». Привязка к участку по КН.
**Нюанс:** число кустов пишут с пробелами («40 931») — парсер это понимает.

### Шаг 3. Техкарта: уход и средства защиты
**Задача:** учесть плановые работы (вспашка, подвязка, обработки) и действующие
вещества (Полирам, Купроксат…) с нормой на га.
**Решение:** парсер техкарты (только виноградники — по вашему указанию) разбирает
листы сметы и СЗР → события поля.
**Проблема:** строки-заголовки сметы попадали в операции; смешивались удобрения и
пестициды. → **Закрыто:** фильтрация по наличию цены; пестициды и удобрения
разделены.

### Шаг 4. Погода: накопленные условия с момента посадки
**Задача:** учесть климат/погоду как ценообразующий фактор урожая.
**Решение:** бесплатный Open-Meteo по геоточке контура → сумма активных температур
(GDD), осадки, радиация, ветер с года посадки.
**Ограничение:** в закрытом контуре сети нет — ядро считается офлайн, реальные
числа подтянутся при наличии интернета.

### Шаг 5. Оценочный профиль (всё вместе)
**Задача:** собрать в одну строку то, что формирует стоимость насаждения.
**Решение:** вьюха **контур ЗУ (площадь/геоточка) × насаждение (сорт/возраст/кусты)
× уход (операции/обработки) × погода (GDD/осадки)**.

### Шаг 6. Ваши правки (ЭТП-профиль)
**Задача:** ваши экспертные данные (износ, отделка, риски) не должны теряться при
автообогащении из реестров.
**Решение:** единое слияние с приоритетом **ручной ввод > ОСВ > NSPD/EXIF/реестры**.
**Проблема:** автоисточник (NSPD) затирал ваш ручной ввод. → **Закрыто:** приоритет —
NSPD только заполняет пустое, ваши значения сохраняются.

### Шаг 7. Лот
**Задача:** собрать состав лота (земля + винодельня) без ручного перечисления.
**Решение:** отбор правилами include/exclude (по КН, по маске квартала, по типу),
результат детерминирован (повтор даёт тот же состав).

### Шаг 8. Bundle (пакет обмена)
**Задача:** отдать бэкенду один воспроизводимый пакет.
**Решение:** каталог `project.kmz` + `db.sqlite` + `json/` + `manifest.json`
(контрольные суммы, версии). Повторная сборка на тех же входах = те же файлы.
**Проблема:** имена файлов с `:` (как в КН) недопустимы в Windows. → **Закрыто:**
файлы объектов пишутся в безопасной маске (`23_15_0804000_66.json`).

## C2-экспорт — ✅ СДЕЛАНО (ADR-007, 2026-06-09)

**Решено:** C2 — главный формат. Написан **слой экспорта** внутренней БД парсера в
C2-форму. Теперь Bundle может нести правильную БД одной командой:
```powershell
egrn-parser bundle --out _bundle --kmz project.kmz --db рабочая.sqlite --lot-id lot-1 --export-c2
```
или отдельно: `egrn-parser export-c2 --db рабочая.sqlite --out c2.sqlite`.
Правила перевода таблиц — `obsidian/Decisions/ADR-007-pkg-schema-to-c2-export.md`.

---

### (Историческая справка) что это было

**Простыми словами.** В Bundle кладётся `db.sqlite`. Внутри парсер работает в
**своём** формате, а контракт обмена с бэкендом (C2) требует БД в **каноническом**
виде (единая таблица объектов, ЭТП-слой и т.д.). Это два разных «языка» базы —
теперь между ними есть переводчик (слой экспорта).

**Что нужно решить (это и есть ADR — короткий документ-решение):**
1. Подтвердить, что **C2 — главный формат** для бэкенда (а внутренний формат парсера
   остаётся только для работы парсера).
2. Согласовать **правила перевода** (mapping): какая внутренняя таблица → в какую
   таблицу C2 (например, внутренние `building_objects` + `land_objects` → единая
   `objects` контракта).
3. Дать добро на написание **слоя экспорта** (parser → C2-`db.sqlite`), чтобы в
   Bundle попадала уже правильная БД.

**Почему без вас нельзя:** это меняет контракт с командой бэкенда/вьюера — нужно ваше
решение, что C2 канон, и сверка правил перевода с ними. Подробный разбор расхождений
готов: `obsidian/Architecture/schema-pkg-vs-c2-drift.md`.

**Что я сделаю после вашего «да»:** напишу ADR с правилами перевода + слой экспорта
`§1–§5 → C2`, и тогда `egrn-parser bundle` будет класть в пакет уже C2-совместимую БД.
