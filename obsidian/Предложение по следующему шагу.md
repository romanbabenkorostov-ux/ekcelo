# Предложение по следующему шагу

Разработка будет вестись независимо разными разработчиками в несколько этапов. Поэтому по каждому направлению (**парсинг** и **визуализация**) требуется не просто ТЗ, а полноценный **spec с чеклистом**.

**Дополнительные требования к рефакторингу:**

- Изучить вложения.
- Отрефакторить парсинг и структуризацию данных в БД и файлах.
- Ограничения объекта как актива структурировать в виде: «Сведения о том, что земельный участок полностью расположен в границах зоны с особыми условиями использования территории, территории объекта культурного наследия, сельскохозяйственного угодья в составе земель сельскохозяйственного назначения, публичного сервитута».
- Ограничения как пассивы: «Ограничение прав и обременение объекта недвижимости» (может быть несколько по разным основаниям).
- Данные из поля «Сведения о возможности предоставления третьим лицам персональных данных физического лица» заменять на хэш uuid.
- При необходимости обновить словари.

**Новые поля (добавить в парсинг и формирование файлов для ОКС и сооружений):**

- «Этажность по документу без подземных» (целое число)
- «Этажей подземных» (целое число)
- «Этажность по осмотру» (текст, заполняется вручную)
- «Состояние по осмотру» (текст, заполняется вручную)

Для помещений/квартир: по полю «Кадастровые номера иных объектов недвижимости, в пределах которых расположен объект недвижимости» определять, является ли родительским объектом не земельный участок, и при необходимости подтягивать этажность ОКС/строения.

Проверить в ТЗ на модуль визуализации наличие чёткого разделения на **права** и **ограничения** (отрефакторить при неконсистентности).

# Задача

Доработать логику работы с файлами, провести рефакторинг и выдать **консистентные ТЗ/spec** для независимой параллельной работы разных программистов, не общающихся между собой.
