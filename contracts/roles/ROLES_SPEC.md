# C6 — Роли и шеринг (контракт; реализация — после веб-шва)

> Зафиксировано **сейчас**, реализуется отдельной итерацией после приземления
> REST-шва (C4). Наследует существующий токен-гейт (`tokens.js`) и Basic Auth
> (cycle 12). OAuth/JWT — будущий триггер.

## Субъекты (роли)

| Роль | Может | Делегирование / передача |
|------|-------|--------------------------|
| **superadmin** | всё: прямой ввод данных, любые лоты/объекты, выгрузки | — |
| **assessor** (партнёр EKCELO) | поиск клиентов, ввод/правка данных по своим объектам/лотам, заказ ввода | **может делегировать** полномочия другим асессорам (агентам) |
| **client** (клиент партнёра) | просмотр готовых данных по предоставленным лотам/объектам, выгрузка Bundle | **может передать право просмотра** третьим лицам |

Заказчик системы = assessor или client (возможен прямой заказ ввода у superadmin).

## Объекты доступа

`lot` · `object` · `bundle` (+ его части kmz/db/json/raw).

## Действия

`input` (ввод) · `edit` (правка §6) · `view` (просмотр ViewModel) ·
`export` (скачивание Bundle/части) · `delegate` · `share`.

## Модель делегирования и расшаривания

- **assessor → assessor**: грант роли `assessor` на подмножество лотов/объектов
  (scoped delegation), с возможностью отзыва.
- **client → третье лицо**: грант `view` (read-only) на конкретные лоты/объекты;
  реализуется как **share-token** поверх текущего токен-гейта
  (`?t=<base64url>` в `ekcelo-site/viewer`), с TTL и областью видимости.
- Все гранты — в БД (новая таблица `access_grants`, §7, вне ЕГРН-слоя),
  не в §1–§6 (ADR-001 не нарушается).

## Привязка к API (C4)

`securitySchemes.bearerAuth` в `openapi.yaml`; каждый путь получает scope-проверку
по (subject_role, object_id, action). До реализации — эндпоинты открыты под
single-owner/токен-гейт, как сейчас.

## Что НЕ в этой итерации

Реализация ACL, таблица `access_grants`, UI делегирования/передачи (frontend
SPEC трек 7) — отдельный cycle с явным триггером «появились реальные
коллабораторы / отдельные PAT».
